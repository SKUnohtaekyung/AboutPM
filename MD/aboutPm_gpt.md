## 제품 조직과 역할

### 개념 카드 R-01: PM 정체성과 세계관

**0) 한 줄 정의(초등 5학년 수준)**  
PM은 “사람들이 불편해하는 문제를 찾아서, 팀이 같이 해결하도록 방향을 잡아주는 사람”입니다. 

**1) 왜 필요한가(해결하는 문제)**  
제품 개발은 “아이디어”가 아니라 “선택의 연속”입니다. 무엇을 만들지, 왜 지금인지, 성공을 어떻게 정의할지 합의가 없으면 기능은 늘고(스코프 팽창) 결과는 불명확해집니다. PM은 고객 문제와 비즈니스 목표를 연결하고, 디자이너/엔지니어가 실행 가능한 형태로 나눠 팀을 정렬시키는 역할을 맡습니다. 

**2) 언제 쓰나(신호/상황/트리거)**  
문서가 아닌 “방향”이 필요한 순간에 PM의 역할이 선명해집니다. 예:  
- 기능 요청이 쏟아지는데 **우선순위 기준이 없다**  
- “좋아 보이는데”라는 말만 있고 **성공 지표가 없다**  
- 디자인/개발이 각각 맞는데도 **출시 후 성과가 없다**(가치/시장/채널 문제 가능)

**3) 어떻게 쓰나(단계별 절차 + 최소 산출물)**  
최소 산출물 기준으로 PM의 하루를 ‘작동하는 루프’로 만들면 다음 4단계가 핵심입니다.  
- 문제 정의: “누가/무엇을/왜 불편한가”를 1페이지로 요약  
- 목표/지표 설정: 목표(Outcome)와 측정 지표(Metric) 1~3개로 고정  
- 해결책 설계: 가장 작은 실험/기능 단위로 설계(MVP 사고)   
- 학습/반복: 출시→측정→배우고→다음 우선순위 갱신(지속 개선) 

**4) 예시 2개(가상 시나리오)**  
- 예시A(B2C 할 일/우선순위 앱)  
상황: 신규 유저가 ‘할 일 추가’는 하는데 다음날 돌아오지 않는다(7일 리텐션 낮음).  
PM의 선택: “리마인드 푸시”부터 만들기 전에, ‘첫 가치 경험’(Aha)을 “오늘 할 일 3개 선정”으로 정의하고 온보딩에서 30초 안에 3개를 뽑게 만드는 플로우를 설계.  
결과: Activation(첫 가치 경험 도달률) 상승 → 7일 리텐션이 동반 상승.

- 예시B(B2B 협업 대시보드 SaaS)  
상황: 데모 신청은 많은데 실제 사용으로 전환이 낮다(Trial→활성화 전환 병목).  
PM의 선택: “대시보드 템플릿 자동 생성”을 MVP로 만들어, 업종별 샘플 데이터를 즉시 보여주고 첫 10분 안에 ‘보고서 공유’까지 도달하게 설계.  
결과: Activation(첫 보고서 생성률)↑ → Trial 전환율↑ → 지원팀 문의(How-to) 감소.

**5) 실무에서 자주 틀리는 지점(안티패턴 3~5개)**  
- “기획자=요구사항 전달자”로 스스로를 축소(방향/성과 책임을 놓침)   
- 지표 없이 출시(“개선”인지 “변화”인지 판단 불가)  
- 고객(문제)·비즈니스(가치)·기술(현실성) 중 하나만 본다(균형 붕괴)   
- “많은 기능=좋은 제품” 착각(복잡도만 증가)

**6) 체크리스트(복붙용)**  
- [ ] 현재 풀려는 문제를 “사용자 문장”으로 1줄로 썼다(예: “처음 온 유저가 내일 다시 돌아올 이유가 없다”)  
- [ ] 목표(Outcome)와 지표(Metric)를 1~3개로 고정했다  
- [ ] 이번 주에 할 일은 “실험/학습” 단위로 쪼갰다  
- [ ] 출시 후 ‘측정→학습→다음 우선순위’ 루프가 캘린더에 잡혀 있다

**7) 관련 지표/측정 방법**  
- Activation/Retention/Conversion(퍼널/AARRR 연결)   
- 정성 지표: 인터뷰에서의 “가치 인식 문장”, UX 테스트에서의 과업 성공률 

**8) 참고자료(인용 포함)**  


### 개념 카드 R-02: PM·PO·PJM 역할 구분

**0) 한 줄 정의**  
PM은 “무엇을/왜 만들지”를 책임지고, PJM은 “언제/어떻게 끝낼지”를 책임지며, PO는(특히 Scrum에서) “제품 가치 최대화”에 대해 공식적으로 책임지는 역할입니다. 

**1) 왜 필요한가**  
세 역할이 섞이면, 팀은 서로 다른 게임을 합니다.  
- PM이 제품 지표를 만들려는데 PJM이 납기만 쫓으면 품질/학습이 무너지고  
- PO가 backlog 우선순위를 정해야 하는데 의사결정 권한이 없으면 Scrum이 형식만 남습니다. 

**2) 언제 쓰나**  
- “누가 최종 결정자인가?”가 매번 회의마다 흔들릴 때  
- 일정이 계속 밀리는데, 이유가 “요구 바뀜”인지 “추정 실패”인지 불명확할 때  
- backlog가 있는데도 “가치 우선순위”가 없을 때

**3) 어떻게 쓰나(절차 + 최소 산출물)**  
- 역할을 ‘직무명’이 아니라 **결정/책임 단위**로 정의한다  
- 최소 산출물: RACI(Responsible/Accountable/Consulted/Informed) 1장 + 성공 지표 표 1장  
- 특히 Scrum을 사용한다면 “PO Accountable” 조건(가치 최대화, backlog 관리)을 충족하는지 점검한다. 

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “다음 달 내 출시”가 목표인데 기능이 계속 늘어난다.  
PM의 선택: PM은 ‘핵심 가치’ 기준으로 In/Out을 고정하고, PJM은 일정·리스크 관리(의존성/리소스)로 스코프를 보호.  
결과: 납기 준수율↑, 출시 후 Activation 측정 가능.

- 예시B(B2B)  
상황: Scrum을 한다고 하지만 backlog 우선순위가 영업/CEO/CS마다 다르다.  
PM의 선택: PO(또는 PO 책임을 가진 사람)를 명확히 지정하고, backlog ordering 기준을 “수익 임팩트/리스크/학습가치”로 합의.  
결과: Sprint 목표 달성률↑, 재작업↓.

**5) 안티패턴**  
- “PO=PM보다 상위 직급”처럼 직급 개념으로 오해(조직마다 다름)   
- PJM이 제품 우선순위를 결정(가치보다 일정 최적화로 흐름)  
- PM이 일정·리소스 관리를 완전히 포기(실행력이 무너짐)

**6) 체크리스트**  
- [ ] “가치(제품 지표)”에 최종 책임(Accountable)인 사람이 누구인지 합의했다   
- [ ] “일정/범위/리스크”를 관리하는 주체가 운영 루틴을 갖고 있다   
- [ ] 의사결정이 충돌할 때 우선순위 규칙(지표/전략)이 문서로 있다

**7) 관련 지표/측정**  
- 제품 지표: Activation/Retention/LTV, MAU 등   
- 프로젝트 지표: Velocity, Burndown, 납기 준수 

**8) 참고자료**  


### 개념 카드 R-03: 조직 구조와 일하는 방식(기능 조직 vs 목적 조직)

**0) 한 줄 정의**  
기능 조직은 “기획/디자인/개발이 부서별로 릴레이”하고, 목적 조직은 “한 팀이 처음부터 끝까지 함께 책임”지는 방식입니다. 

**1) 왜 필요한가**  
조직 구조는 “문서 양”과 “회의 방식”을 결정합니다.  
기능 조직은 정렬을 위해 문서가 두꺼워지기 쉽고, 목적 조직은 빠른 실험을 위해 짧지만 빈번한 정렬이 중요합니다. (둘 다 장단점이 있습니다.) 

**2) 언제 쓰나**  
- “왜 이렇게 문서가 많아?” 혹은 “왜 이렇게 말로만 해?” 갈등이 생길 때  
- 요구사항 변경이 잦은데 Waterfall로 운영해 계속 ‘막판 폭발’이 날 때 

**3) 어떻게 쓰나**  
- 먼저 “변경 가능성(불확실성)”을 평가한다: 클수록 Agile/목적 조직의 이점이 커진다.   
- 기능 조직에서는 명시적 산출물(IA/Use Case/PRD)로 손실을 줄이고  
- 목적 조직에서는 “짧은 PRD + 프로토타입/UT”로 학습 속도를 높인다.

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 광고 캠페인에 맞춰 2주 안에 신규 온보딩 개선이 필요.  
PM의 선택: 크로스펑셔널 스쿼드로 2주 스프린트 구성(디자이너+개발자 상시).  
결과: 리드타임↓, Activation↑.

- 예시B(B2B)  
상황: 대기업 고객 커스터마이징 프로젝트, 요구사항이 많은 대신 변경은 상대적으로 적음.  
PM의 선택: 기능 조직+PJM을 세워 범위/납기/승인 프로세스를 견고히.  
결과: 납기 준수↑, 리워크↓.

**5) 안티패턴**  
- Waterfall을 쓰면서 “변경은 언제든 가능”하다고 약속(막판 폭발)   
- Agile을 쓰면서도 의사결정 권한이 분산(PO 부재, backlog 혼란)   
- 목적 조직인데 산출물 최소화가 아닌 “산출물 부재”로 흘러 커뮤니케이션 비용 증가

**6) 체크리스트**  
- [ ] 변경 가능성(불확실성) 수준에 맞춰 프로세스를 선택했다  
- [ ] 어떤 산출물은 “필수”이고(예: 성공지표/범위), 어떤 것은 “필요할 때만”인지 합의했다  
- [ ] 역할(결정권)과 루틴(주간/일간 정렬)을 명확히 했다

**7) 관련 지표**  
- 리드타임(아이디어→출시), 재작업률, 결함률, 고객 불만(VOC) 처리시간

**8) 참고자료**  


### 개념 카드 R-04: 채용 공고로 역량 역추적

**0) 한 줄 정의**  
채용 공고는 “회사들이 PM에게 실제로 돈 주고 사고 싶은 능력”의 목록입니다. 

**1) 왜 필요한가**  
초심자는 “무엇부터 공부해야 할지” 막힙니다. 채용 공고의 “주요 업무/자격 요건”을 분해하면 PM 역량이 **문서·데이터·협업·도메인**으로 수렴합니다. 

**2) 언제 쓰나**  
- 커리어 전환/주니어 입문 시  
- 팀 내에서 “PM 역할 기대치”를 맞추고 싶을 때

**3) 어떻게 쓰나**  
- 공고 10개를 모아 공통 요구사항을 분류한다(문서화/데이터/커뮤니케이션/실행)  
- 최소 산출물: “역량 매트릭스(행=역량, 열=증빙 산출물)” 1장  
- 역량을 증빙할 산출물(PRD, IA, Use Case, UT 리포트 등)을 포트폴리오로 묶는다.

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “데이터 기반 개선”이 공통 요구인데 경험이 없다.  
PM의 선택: 실제 앱을 기준으로 AARRR 정의→이벤트 설계→가설 1개→간이 UT 5명→PRD 업데이트까지를 포트폴리오로 제작.  
결과: “데이터/실험/문서” 역량이 한 번에 증빙된다.

- 예시B(B2B)  
상황: “B2B 요구사항 정리”가 공고에서 반복.  
PM의 선택: Use Case + IA + 권한(roles) + 감사로그 요구사항까지 포함한 PRD 샘플 제작.  
결과: 현업형 문서화/리스크 인식이 드러난다.

**5) 안티패턴**  
- 공고의 “툴 이름”만 외우고, 실제 산출물/사고 과정이 없다  
- 포트폴리오가 UI만 예쁘고 “문제/지표/성과”가 비어 있다  
- “협업”을 말로만 쓰고, 실제 커뮤니케이션 산출물이 없다(회의록/결정 로그 등)

**6) 체크리스트**  
- [ ] 공고 10개 이상에서 반복되는 역량 5개를 뽑았다  
- [ ] 각 역량을 증빙할 산출물을 1개 이상 만들었다  
- [ ] 산출물에 “상황→선택→지표 변화” 이야기가 포함된다

**7) 지표/측정**  
- 포트폴리오 자체의 평가 기준: 문제 정의 명확성, 지표 적합성, 실행 가능성, 학습 루프 증빙

**8) 참고자료**  
(원문 기반 커리큘럼 해석) 

### 장 마무리: 실무 체크리스트와 안티패턴

이 장(제품 조직과 역할)의 핵심은 “직무명”보다 “결정과 책임”입니다.

실무 체크리스트  
- [ ] ‘가치(제품지표)’와 ‘일정(프로젝트지표)’를 같은 사람이 동시에 최적화하려고 하지 않는다   
- [ ] 조직 구조에 맞는 문서/회의 루틴을 설계한다   
- [ ] “결정 로그(Decision log)”를 남긴다(왜 이걸 선택했는지)

자주 하는 실수(안티패턴)  
- 역할을 모호하게 둔 채 “협업”만 강조(결정 충돌이 계속된다)  
- Agile을 “문서 없음”으로 착각(실제로는 정렬이 더 자주 필요하다)

## 고객 이해와 UX 설계

### 개념 카드 U-01: 페르소나

**0) 한 줄 정의**  
페르소나는 “우리 제품의 대표 사용자 한 명을 실제 사람처럼 구체적으로 그린 가짜 인물”입니다. 

**1) 왜 필요한가**  
“모든 사용자”를 만족시키려 하면 아무도 만족시키지 못합니다. 페르소나는 팀이 같은 사용자를 떠올리며 우선순위를 정하게 하고, 기능/문구/UX 결정의 기준이 됩니다. 

**2) 언제 쓰나**  
- 타겟이 넓어서 기능이 산으로 갈 때  
- 팀 내에서 “이 기능 누가 쓰는데?”가 반복될 때  
- 온보딩/핵심 플로우를 설계할 때

**3) 어떻게 쓰나(단계 + 최소 산출물)**  
- 리서치 기반 단서 모으기(인터뷰/설문/로그/VOC)  
- 사용자 군집(패턴) 2~3개로 묶기  
- 대표 페르소나 1~2개 작성(초심자는 1개부터)  
- 최소 산출물: 페르소나 카드 1장(목표/상황/니즈/페인포인트/대체재)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 투두 앱이 “학생/직장인/프리랜서” 모두를 겨냥해 기능이 복잡해짐.  
PM의 선택: “마감이 있는 업무를 놓치면 큰 손해”인 직장인을 대표 페르소나로 고정하고, 우선순위를 ‘마감 기반 정렬’로 통일.  
결과: 첫 주 과업 완료율↑, 리텐션↑.

- 예시B(B2B)  
상황: 대시보드 제품에 기능요청이 난립.  
PM의 선택: “보고서를 상사에게 매주 제출해야 하는 운영 매니저” 페르소나를 기준으로, ‘자동 리포트/공유’에 집중.  
결과: 활성 계정당 공유 횟수↑, 유료 전환↑.

**5) 안티패턴(3~5개)**  
- Bad: “20대 후반 스타트업 직원”처럼 너무 넓다(결정 기준이 안 된다).   
- Good처럼 보이지만 실제로는 ‘소설’: 리서치 근거가 없다  
- 페르소나가 “데모용 PPT”로 끝나고 backlog/PRD에 반영되지 않는다

**6) 체크리스트(복붙용)**  
- [ ] 페르소나에 “목표(Goal)”가 포함되어 있다  
- [ ] ‘대체재(지금 쓰는 방식/경쟁 서비스)’가 있다  
- [ ] 페인포인트가 3개 이하로 정리되어 있다  
- [ ] 기능 우선순위를 페르소나 기준으로 설명할 수 있다

**7) 관련 지표/측정**  
- 페르소나별 코호트 리텐션, 전환율 차이(가능하면 세그먼트로 본다)  
- 정성: 인터뷰에서 “이게 딱 내 문제”라는 발화 비율

**8) 참고자료**  


#### Persona Bad vs Good 예시(샘플)

Bad(나쁜 페르소나)  
- “20대 후반의 스타트업 직원”  
→ 너무 넓어 기능 결정이 안 된다(‘누구’인지가 아니라 ‘상황/목표/제약’이 비어 있음).   

Good(좋은 페르소나)  
- 이름: 정근(27)  
- 상황: 출퇴근 지하철에서 하루 계획을 빠르게 잡아야 함  
- 목표: “오늘 꼭 해야 하는 3개만 지키기”  
- 제약: 회사 일정이 자주 바뀌어 계획이 깨짐  
- 대체재: 메모앱 + 캘린더 + 머릿속 기억  
- 페인포인트: 우선순위가 계속 뒤집힘, 미완료가 쌓여 죄책감  
→ “어떤 UX가 도움이 되는지”가 바로 떠오른다(예: 3개 고정, 일정 변경 시 자동 재정렬).

### 개념 카드 U-02: 데스크 리서치(시장 조사 포함)

**0) 한 줄 정의**  
데스크 리서치는 “사람을 만나기 전에, 이미 세상에 있는 자료로 시장과 사용자를 공부하는 것”입니다.

**1) 왜 필요한가**  
인터뷰는 비싸고 느립니다. 데스크 리서치로 기본 가설(누가/무엇을/어떤 대안으로 해결 중인지)을 세우면, 인터뷰 질문이 날카로워지고 실수(엉뚱한 시장 진입)가 줄어듭니다.

**2) 언제 쓰나**  
- 제품 아이디어 초기(Discover 단계)   
- 경쟁/대체재가 많은 시장에서 포지셔닝이 불명확할 때  
- 인터뷰 질문 설계 전

**3) 어떻게 쓰나**  
- 시장: 시장 규모/성장률/규제/핵심 플레이어  
- 사용자: 리뷰/커뮤니티/FAQ/검색어(문제 표현)  
- 제품: 경쟁사 온보딩/유료벽/핵심 가치 제안  
- 최소 산출물: “요약 1장 + 질문 10개” (인터뷰 가이드로 연결)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 투두 앱이 포화.  
PM의 선택: 앱스토어 리뷰에서 반복되는 불만(“우선순위가 어렵다”)을 추출해 ‘우선순위 자동화’가 틈새인지 검증.  
결과: 인터뷰 질문이 “기능 아이디어”가 아닌 “문제 강도/대체재” 중심으로 정교해짐.

- 예시B(B2B)  
상황: 대시보드 SaaS 경쟁이 치열.  
PM의 선택: 경쟁사 가격/권한/감사로그 유무를 비교표로 만들고, “보안/컴플라이언스”가 차별점이 될지 가설 수립.  
결과: 세일즈 메시지/PRD의 성공지표가 명확해짐.

**5) 안티패턴**  
- 자료를 많이 모았는데 결론이 없다(“그래서 우리가 뭘 할 건데?”)  
- “시장 규모”만 보고 사용자 문제를 안 본다  
- 경쟁사 기능을 복사하는 데스크 리서치(차별화가 사라짐)

**6) 체크리스트**  
- [ ] 사용자 문제를 ‘사용자 언어’로 10개 수집했다  
- [ ] 대체재(경쟁/비경쟁 포함)를 최소 5개 분석했다  
- [ ] 인터뷰에서 검증할 가설을 3개로 줄였다

**7) 지표/측정**  
- 데스크 리서치 자체는 지표가 아니라 “가설의 품질”로 평가  
- 후속 지표: 인터뷰에서 가설이 얼마나 빠르게 반박/강화되는지(learn cycle 속도)

**8) 참고자료**  
(프로세스 맥락은 더블 다이아몬드의 Discover 단계와 연결) 

### 개념 카드 U-03: 고객 인터뷰/IDI

**0) 한 줄 정의**  
고객 인터뷰는 “진짜 사용자가 왜 그렇게 행동하는지, 말로 들어보는 것”입니다.

**1) 왜 필요한가**  
로그는 “무슨 일이 일어났는지”는 알려주지만 “왜”는 끝까지 알려주지 않습니다. 인터뷰/IDI는 동기·맥락·대체재를 밝혀 PRD의 ‘왜’(문제 정의)를 단단하게 만듭니다. 

**2) 언제 쓰나**  
- 지표가 떨어졌는데 원인이 여러 개일 때  
- 새로운 가치 제안/가격/온보딩을 바꿀 때  
- VOC가 서로 충돌할 때(누구의 문제인지 분해 필요)

**3) 어떻게 쓰나**  
- 목적을 정한다: 탐색(문제 발견) vs 검증(가설 테스트)  
- 질문은 “기능” 대신 “상황/행동/대체재/결정 기준” 중심으로 만든다  
- 최소 산출물: 인터뷰 노트 → 인사이트 5개 → “PRD에 반영할 문장 3개”

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 유저가 알림을 꺼버림.  
PM의 선택: “왜 알림이 싫었나”를 상황 질문으로 파고들어(언제/무슨 이유로) 맞춤 알림 규칙을 설계.  
결과: 알림 차단율↓, 재방문↑.

- 예시B(B2B)  
상황: 팀원이 제품을 쓰지 않는다(시트로 회귀).  
PM의 선택: “업무 흐름 속 ‘인수인계’ 순간”에서 제품이 끊긴다는 걸 발견 후, 공유/권한/댓글 기능을 우선 개발.  
결과: 주간 활성 사용자(Team WAU)↑, 시트 의존↓.

**5) 안티패턴**  
- “이 기능 필요하세요?”처럼 유도 질문  
- 사용자가 말한 해결책을 그대로 구현(문제와 해결책을 혼동)  
- 인터뷰 후 “인사이트”가 아니라 “감상문”만 남는다

**6) 체크리스트**  
- [ ] 인터뷰 목적(탐색/검증)을 한 문장으로 썼다  
- [ ] 대체재(현재 어떻게 해결하는지)를 반드시 물었다  
- [ ] “결정 기준”을 물었다(왜 A를 선택하고 B를 포기했나)  
- [ ] 인사이트를 PRD 문장으로 바꿔 적었다

**7) 지표/측정**  
- 인터뷰 품질: 재현 가능한 패턴(반복되는 상황/행동)이 나오는지  
- 후속 지표: 개선 후 과업 성공률, 리텐션 변화

**8) 참고자료**  
사용성 테스트와의 차이를 포함해 UX 리서치의 주요 방법으로 널리 설명됨. 

### 개념 카드 U-04: VOC 운영

**0) 한 줄 정의**  
VOC는 “고객이 남긴 불만/요청/칭찬을 모아, 제품 개선 순서를 정하는 재료”입니다.

**1) 왜 필요한가**  
VOC를 ‘좋은 말/나쁜 말’로만 보면, 제품이 “소음에 끌려다니는” 상태가 됩니다. VOC를 분류·정량화·우선순위화해야 PM이 ‘학습’으로 전환할 수 있습니다.

**2) 언제 쓰나**  
- 고객문의/리뷰가 늘어나는데 해결이 반복될 때  
- 리텐션/전환이 떨어지는데 행동 로그만으로 원인이 모호할 때

**3) 어떻게 쓰나**  
- 카테고리(온보딩/결제/성능/콘텐츠/권한 등)로 태깅  
- 심각도(빈도×임팩트×대체 가능성)로 우선순위  
- 최소 산출물: VOC Top10 + “PRD 반영 여부/시점” 표

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “우선순위가 헷갈려요” VOC 급증.  
PM의 선택: VOC를 ‘결정 피로’ 카테고리로 묶고, 첫 화면에서 3개 추천+수정 방식으로 단순화.  
결과: 첫 주 이탈↓, 리뷰 평점↑.

- 예시B(B2B)  
상황: “누가 뭘 봤는지 모르겠다” VOC 반복.  
PM의 선택: 감사로그/권한 요구를 한 덩어리로 정의하고, 엔터프라이즈 플랜의 핵심 가치로 PRD에 반영.  
결과: 업셀 전환↑, 이탈(해지) 감소.

**5) 안티패턴**  
- VOC 수집만 하고 제품 의사결정으로 연결되지 않는다  
- “큰 고객” 의견만 반영해 제품이 왜곡된다  
- VOC를 기능 리스트로만 취급(근본 원인을 놓침)

**6) 체크리스트**  
- [ ] VOC는 태그/카테고리로 분류된다  
- [ ] 우선순위 기준이 지표(리텐션/전환/매출)와 연결된다  
- [ ] VOC→가설→실험→결과의 추적이 가능하다

**7) 지표/측정**  
- VOC 처리시간, 동일 이슈 재발률, CS 티켓 감소율  
- 제품 지표: 리텐션/전환 개선으로 연결되는지

**8) 참고자료**  
(원문 기반 운영 개념 확장) 

### 개념 카드 U-05: 유저 저니 맵

**0) 한 줄 정의**  
저니 맵은 “사용자가 목표를 이루는 동안 거치는 과정을 시간순으로 그린 지도”입니다. 

**1) 왜 필요한가**  
저니 맵은 ‘화면’이 아니라 ‘경험’을 다룹니다. 어디에서 감정이 떨어지고(좌절), 어디에서 가치가 생기며(만족), 어디가 병목인지(이탈)를 한눈에 보여줍니다. 

**2) 언제 쓰나**  
- “기능은 많은데 왜 안 쓰지?”가 반복될 때  
- 온보딩/결제/공유 같은 핵심 여정에서 이탈이 클 때  
- 여러 채널(웹/앱/메일/CS)이 얽혀 문제 원인이 분산될 때

**3) 어떻게 쓰나(단계 + 최소 산출물)**  
- 페르소나를 고른다(누구의 여정인가)  
- 단계 정의(인지→유입→탐색→행동→재방문 등)  
- 각 단계에서 행동/생각/감정을 정성 데이터로 채운다  
- Pain Point와 기회요인을 “가설”로 만든다  
- 최소 산출물: 저니 맵 1장 + 개선 가설 3개

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 유입은 많은데 가입 후 첫 과업 완료가 낮다.  
PM의 선택: 저니 맵에서 “가입 직후” 감정이 급락(정보 입력 피로)하는 것을 발견 → 입력을 1단계로 줄이고 나머지는 사용 중에 채우도록 변경.  
결과: 가입→첫 과업 완료 전환율↑, Activation↑.

- 예시B(B2B)  
상황: 사용자가 첫 대시보드를 만들기는 하지만 팀 공유를 안 한다.  
PM의 선택: “팀 공유” 단계에서 ‘권한 설정’이 불안(누가 볼까)하다는 감정 하락을 발견 → 기본 권한 정책/미리보기/감사로그 안내를 추가.  
결과: 공유 기능 사용률↑, 팀 단위 리텐션↑.

**5) 안티패턴**  
- ‘그림만 예쁜’ 저니 맵(리서치 근거가 없다)   
- 단계가 UI 화면 순서로만 구성되어, 감정/결정 포인트를 놓침  
- Pain Point를 “개선 아이디어”로 바로 점프(가설 검증이 없다)

**6) 체크리스트(복붙용)**  
- [ ] 저니 맵은 “목표 1개” 중심이다  
- [ ] 행동(Action)과 감정(Emotion)이 분리되어 있다  
- [ ] 최저점(골짜기) 1~2개가 명확하다  
- [ ] 골짜기마다 “원인 가설”이 있다

**7) 관련 지표/측정**  
- 단계별 전환율(퍼널), 과업 성공률(UT), 이탈률  
- 감정은 정량화 어려우므로 “대리지표”(예: 중간 이탈률, 문의 발생률)을 사용

**8) 참고자료**

| 단계 | 사용자 행동 | 사용자 감정(예) | 페인포인트(가설) | 기회(실험 아이디어) |

| 습관화 | 주간 계획 | 🙂 만족 | 일정 변화에 취약 | 자동 재정렬/캘린더 연동 |

- [ ] 정성(인터뷰/VOC)→정량(지표) 연결 고리를 PRD로 문서화한다

## 데이터 기반 성장

### 개념 카드 G-01: 퍼널

**0) 한 줄 정의**  
퍼널은 “사람들이 목표 행동을 하기까지, 단계별로 얼마나 남고 얼마나 떨어지는지 보는 방법”입니다. 

**1) 왜 필요한가**  
전환 문제는 “전체적으로 나쁨”이 아니라 “특정 단계에서 무너짐”인 경우가 많습니다. 퍼널은 병목 지점을 빠르게 찾아, 제한된 리소스로 가장 큰 임팩트를 만들게 합니다. 

**2) 언제 쓰나**  
- 가입/결제/첫 과업 완료처럼 목표 행동이 뚜렷할 때  
- 마케팅 유입은 늘었는데 매출이 안 늘 때  
- UX 변경 후 효과를 확인할 때

**3) 어떻게 쓰나**  
- 목표 행동을 정한다(예: 결제 완료)  
- 그 목표까지의 필수 단계를 4~7개로 정의한다  
- 각 단계의 전환율과 이탈률을 계산한다  
- 최소 산출물: 퍼널 정의 표(단계/이벤트/분모/분자) 1장

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 가입은 늘었는데 유료 전환이 낮다.  
PM의 선택: 퍼널을 “가입→첫 리스트 생성→7일 내 5회 사용→유료 제안 노출→결제”로 쪼개니 ‘유료 제안 노출’ 이전에 이미 이탈.  
결과: 가격이 아니라 ‘핵심 가치 경험’이 병목임을 확인 → Activation 개선이 우선.

- 예시B(B2B)  
상황: 데모 신청은 높은데 계약이 낮다.  
PM의 선택: “데모 신청→세일즈 미팅→POC 세팅→팀 초대→주간 활성→계약” 퍼널에서 ‘팀 초대’가 병목.  
결과: 권한/초대 UX 개선으로 팀 확장이 빨라져 계약 전환↑.

**5) 안티패턴**  
- 단계를 너무 많이 쪼갬(해석 지옥)  
- 단계를 너무 크게 잡음(병목이 안 보임)  
- “전환율”만 보고 질(리텐션/만족)을 무시

**6) 체크리스트**  
- [ ] 단계는 ‘반드시 필요한 행동’으로만 구성됐다  
- [ ] 각 단계는 이벤트로 측정 가능하다  
- [ ] 병목 1개를 골라 실험 1~3개를 설계했다

**7) 지표/측정**  
- 단계별 전환율, 이탈률, 단계별 체류시간  
- 세그먼트별 퍼널(신규/복귀, 채널별, 플랜별 등)

**8) 참고자료**  


### 개념 카드 G-02: AARRR

**0) 한 줄 정의**  
AARRR는 “사용자가 들어오고(획득), 가치 느끼고(활성화), 계속 쓰고(유지), 추천하고(추천), 돈 내는(수익) 흐름”을 단계로 보는 틀입니다. 

**1) 왜 필요한가**  
AARRR는 ‘측정 포인트’를 제공해, 성장 논의를 “감”이 아니라 “어느 단계가 문제인지”로 바꾸게 합니다. 특히 초기 제품은 모든 것을 한꺼번에 고치기 어려우므로, 단계별로 우선순위를 잡는 데 유용합니다. 

**2) 언제 쓰나**  
- 성장(유입/전환/리텐션/매출) 문제를 구조화하고 싶을 때  
- 팀이 ‘마케팅’과 ‘제품’의 경계를 두고 싸울 때(사실은 같은 퍼널)  
- 지표 체계를 처음 세팅할 때

**3) 어떻게 쓰나(절차 + 최소 산출물)**  
- 우리 제품의 “목표 행동”을 정한다(예: 주 3회 과업 완료 / 팀 공유 2회)  
- 각 단계의 대표 지표를 1~2개로 고른다  
- 병목 단계 1개를 골라 2주 단위 실험을 한다  
- 최소 산출물: AARRR 대시보드(단계별 1~2 지표) + 실험 로그

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: Acquisition은 되는데 Retention이 낮다.  
PM의 선택: Retention 병목으로 보고, “주간 목표 리포트”와 “연속 달성 배지” 실험을 설계.  
결과: 7일/30일 리텐션↑, DAU/MAU↑.

- 예시B(B2B)  
상황: Activation이 낮아 ‘첫 대시보드 생성’에 실패.  
PM의 선택: ‘템플릿 자동 생성’ + ‘샘플 데이터’로 Time-to-Value를 줄이는 MVP를 출시.  
결과: Activation(첫 대시보드 생성률)↑, Trial 전환↑.

**5) 안티패턴**  
- 모든 단계 지표를 동시에 최적화하려다 아무것도 못함  
- Acquisition만 보고 제품 문제를 외면(유입을 늘려도 샌다)  
- AARRR를 “마케팅 프레임워크”로만 오해(제품 UX가 핵심) 

**6) 체크리스트(복붙용)**  
- [ ] 각 단계의 대표 지표가 1~2개다  
- [ ] 지금 분기에는 병목 단계 1개에 집중한다  
- [ ] 실험마다 “가설→변경→예상 영향→실제 결과”가 기록된다

**7) 관련 지표/측정 방법(표)**

| 단계 | 대표 지표(예) | 병목 진단 질문 | 개선 실험 아이디어(예) |
|---|---|---|---|
| Acquisition | 신규 유입, 채널별 CAC | 어떤 채널이 ‘질 좋은 유입’인가? | SEO/콘텐츠, 추천 유입 루프  |
| Activation | 첫 핵심 행동 완료율, TTV(Time-to-Value) | 사용자는 언제 “아하”를 느끼나? | 온보딩 단축, 템플릿/샘플 데이터 |
| Retention | 7/30일 리텐션, DAU/MAU | 왜 돌아오지 않나(대체재/습관)? | 리마인드, 리포트, 개인화 |
| Referral | 초대/공유율, K-factor | 추천이 ‘의미 있는 보상’인가? | 양방향 보상, 공유 UX 단순화 |
| Revenue | 전환율, ARPA, LTV | 돈 내는 순간의 불안/마찰은? | 가격 실험, 패키징, 무료-유료 경계 조정 |

**8) 참고자료(인용 포함)**  
AARRR의 창시/대중화는  및  맥락에서 “Pirate Metrics”로 알려져 있습니다. 

### 개념 카드 M-01: 핵심 제품 지표 세트(MAU·Retention·Churn·LTV)

**0) 한 줄 정의**  
이 지표들은 “우리 제품이 사람들에게 계속 가치가 있고, 그래서 돈도 벌 수 있는지”를 숫자로 보는 방법입니다. 

**1) 왜 필요한가**  
- MAU는 “얼마나 많은 사람이 실제로 쓰는지”   
- Retention/Churn은 “얼마나 오래 남는지”   
- LTV는 “고객이 평생 가져다 줄 가치”   
즉, 단기 유입이 아니라 **지속 가능한 성장**을 판단하는 필수 지표입니다.

**2) 언제 쓰나**  
- 제품 성과를 설명해야 할 때(리더십/투자/세일즈)  
- 어떤 기능이 “진짜 가치”인지 판단할 때  
- 가격/패키징을 바꿀 때(Revenue 단계)

**3) 어떻게 쓰나**  
- “활성(Active)” 정의부터 합의한다(예: 앱을 열었다 vs 핵심 행동을 했다)   
- 리텐션은 코호트로 본다(가입일 기준 그룹 비교)  
- LTV는 간단한 근사치부터 시작한다(ARPA×마진÷이탈률 등)   
- 최소 산출물: 지표 정의서(정의/분모·분자/이벤트/세그먼트/주의점) 1장

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: MAU는 유지되는데 유료 매출이 정체.  
PM의 선택: LTV 세그먼트를 나눠 “고가치 유저”의 행동 패턴(주간 계획 사용)을 발견, 그 행동으로 온보딩을 유도.  
결과: 유료 전환율↑, LTV↑.

- 예시B(B2B)  
상황: 매출은 늘지만 고객사가 빠르게 해지(Churn).  
PM의 선택: 해지 인터뷰 + 사용 로그로 “팀 확산”이 안 되면 해지된다는 패턴 확인 → 초대/권한/템플릿 강화.  
결과: 로고 churn↓, NRR(순매출 유지)↑(조직에 따라 측정).

**5) 안티패턴**  
- MAU를 “열어보기”로 정의해 의미 없는 부풀림  
- 리텐션을 평균으로만 보고 코호트를 안 봄  
- LTV를 계산하려다 데이터/가정이 부족한데도 숫자만 확정(허상)

**6) 체크리스트**  
- [ ] Active 정의가 “핵심 가치 행동”과 연결된다  
- [ ] 리텐션은 코호트로 보고, 비교 축(채널/온보딩 버전)이 있다  
- [ ] LTV는 ‘현재 가정’을 문서로 남긴다(마진/이탈률)   

**7) 관련 지표/측정 방법**  
- MAU: 30일 내 고유 사용자 수(정의에 따라 다름)   
- Churn: 기간 내 이탈 고객(또는 매출)의 비율   
- LTV/CLV(정의): 고객 관계 전체에서의 총 가치/이익   

**8) 참고자료**  


### 장 마무리: 실무 체크리스트와 안티패턴

실무 체크리스트  
- [ ] 퍼널/AARRR는 “지표를 정하기 위한 도구”가 아니라 “우선순위 회의의 공용 언어”로 쓴다   
- [ ] 지표 정의서(Active 정의 포함)를 먼저 만든다   

자주 하는 실수(안티패턴)  
- Acquisition 성과만 강조해 “새는 바가지”를 키움  
- 지표가 많아져 누구도 보지 않는 대시보드가 됨

## 구현과 운영



### 개념 카드 P-01: 더블 다이아몬드

**0) 한 줄 정의**  
더블 다이아몬드는 “문제를 넓게 찾고(확산), 핵심을 정한 뒤(수렴), 해결책을 넓게 만들고(확산), 최종안을 정하는(수렴) 작업 흐름”입니다. 

**1) 왜 필요한가**  
초심자의 흔한 실수는 “바로 해결책부터 만든다”입니다. 더블 다이아몬드는 그 반대로, **문제 정의의 품질**이 결과를 좌우한다는 점을 구조로 강제합니다. 

**2) 언제 쓰나**  
- 신규 기능/신사업처럼 불확실성이 큰 작업  
- UX/전환 문제가 복합적일 때  
- 팀이 “문제”가 아니라 “아이디어”로 싸울 때

**3) 어떻게 쓰나(절차 + 최소 산출물)**  
- Discover: 조사/인터뷰/데스크 리서치로 문제 공간 확장  
- Define: 페르소나/저니 맵으로 핵심 문제 정의  
- Develop: 아이디어/프로토타입/MVP 설계  
- Deliver: UT/데이터로 검증 후 출시(또는 반복)   
최소 산출물은 “문제 정의 1장 + MVP 가설 1장 + 검증 리포트 1장”입니다.

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 유입은 많은데 재방문이 낮다(리텐션 문제).  
PM의 선택: Discover/Define에서 “우선순위 결정 피로”가 핵심임을 찾고, Develop에서 “오늘 3개 추천” MVP를 만들고 UT로 검증 후 Deliver.  
결과: 7일 리텐션↑.

- 예시B(B2B)  
상황: 도입은 되는데 확산이 안 된다.  
PM의 선택: Define에서 “팀이 함께 보는 순간(공유/권한)”이 가치임을 정의, Develop에서 공유 플로우 MVP 제작, Deliver에서 세그먼트별 전환 측정.  
결과: 팀 초대율↑, 해지율↓.

**5) 안티패턴**  
- Define 없이 바로 Develop(해결책 도박)  
- Discover를 “자료 모으기”로 끝내고 통찰이 없다  
- Deliver에서 측정 없이 ‘감’으로 성공 선언

**6) 체크리스트**  
- [ ] Define 단계에서 “핵심 문제 문장”이 있다  
- [ ] MVP는 학습 목표(검증할 가설)가 명확하다   
- [ ] Deliver에서 성공 기준(지표)이 고정돼 있다

**7) 지표/측정**  
- Define 품질: UT에서 과업 실패 원인이 문제 정의와 맞는지  
- Deliver 결과: AARRR 해당 단계의 지표 변화

**8) 참고자료**  
더블 다이아몬드는  에 의해 널리 소개된 디자인/혁신 프로세스입니다. 

#### 더블 다이아몬드 타임라인 예시(2주/4주/8주)

| 기간 | 목표 | Discover 산출물 | Define 산출물 | Develop 산출물 | Deliver 산출물 |
|---|---|---|---|---|---|
| 2주 | 빠른 개선 실험 | VOC/로그 요약 1장 | 핵심 문제 1문장 + 성공지표 | 저충실도 프로토타입 | UT 5명 + 지표 1주 측정 |
| 4주 | 기능 MVP 출시 | 인터뷰 6~8명, 경쟁 분석 | 페르소나 1개 + 저니 맵 | MVP 범위(In/Out) + PRD 초안 | 베타 출시 + 코호트 리텐션 측정 |
| 8주 | 신규 가치 제안 검증 | 시장/세그먼트 조사 | 문제-가치 가설, 포지셔닝 | MVP 2안 비교(A/B) | 정식 출시 판단(Go/No-Go) |

### 개념 카드 P-02: MVP

**0) 한 줄 정의**  
MVP는 “최소한으로 만들고, 최대한 빨리 배우기 위한 첫 버전”입니다. 

**1) 왜 필요한가**  
불확실한 시장에서 ‘완벽한 제품’은 늦고 비쌉니다. MVP는 최소 노력으로 검증 학습을 얻어, 잘못된 방향에 큰 비용을 쓰는 위험을 줄입니다. 

**2) 언제 쓰나**  
- 우리가 풀고자 하는 문제가 “진짜로 존재하는지”가 불확실할 때  
- 특정 기능이 리텐션/매출에 영향이 있을지 모르겠을 때  
- 가격/패키징 같은 민감한 변경 전

**3) 어떻게 쓰나**  
- “검증 가설”을 하나로 고정한다  
- 그 가설을 검증하는 데 필요한 최소 기능만 만든다  
- 측정 이벤트/성공 기준을 먼저 정의한다  
- 최소 산출물: 가설 1문장 + MVP 범위 표 + 측정 계획

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “우선순위 추천”이 리텐션을 올릴지 모름.  
PM의 선택: ML 모델을 만들기 전에 규칙 기반(마감/반복/미완료) 추천으로 MVP 구현.  
결과: 추천 사용률↑, 리텐션 변화로 가설 일부 검증.

- 예시B(B2B)  
상황: “공유 링크”가 구독 전환에 영향?  
PM의 선택: 완전한 권한 시스템 전, 읽기 전용 공유 링크 + 만료 옵션만 MVP로 출시.  
결과: 공유율↑, 계약 전환 퍼널에서 병목 완화.

**5) 안티패턴**  
- MVP를 “저품질 제품”로 오해(최소는 ‘학습’ 기준이지 ‘대충’이 아님)   
- 가설/성공 기준 없이 MVP를 만듦  
- “기술 부채”를 방치해 다음 반복이 느려짐

**6) 체크리스트**  
- [ ] 가설이 “측정 가능한 문장”이다  
- [ ] 만들지 않을 것(Out of scope)을 명시했다  
- [ ] 성공 기준을 지표로 적었다

**7) 지표/측정**  
- Activation/Retention의 변화(가설에 따라)  
- Time-to-Value 변화(첫 가치 도달 시간)

**8) 참고자료**  
MVP 정의는  의 설명(검증 학습 최대화/최소 노력)에서 널리 인용됩니다. 

### 개념 카드 D-02: PRD(샘플 템플릿 포함)

**0) 한 줄 정의**  
PRD는 “만들 제품(기능)이 무엇이고, 왜 만들며, 성공을 어떻게 볼지”를 팀이 합의하는 문서입니다. 

**1) 왜 필요한가**  
PRD는 ‘상세 스펙’이라기보다 **정렬(alignment)** 입니다. 목적/성공기준이 없으면 개발은 진행돼도 “원하는 결과”가 나지 않습니다. 

**2) 언제 쓰나**  
- 개발 착수 전에 “왜/무엇/성공”을 합의해야 할 때  
- 이해관계자(영업/CS/경영진)가 많을 때  
- 여러 팀이 동시에 같은 기능을 만질 때

**3) 어떻게 쓰나(단계 + 최소 산출물)**  
- 문제/목표/성공지표를 먼저 쓰고  
- 사용자/시나리오(Use Case)로 요구를 구체화하고  
- 범위(In/Out)와 리스크를 명시하고  
- 추적(이벤트/로그)과 출시 계획을 넣습니다. 

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 리텐션 낮음.  
PM의 선택: “첫날 3개 완료”를 목표로 PRD 작성 → 온보딩/추천/리마인드 범위를 정함.  
결과: Activation↑, 리텐션↑.

- 예시B(B2B)  
상황: 팀 확산 부족.  
PM의 선택: “공유/권한” 기능 PRD로 범위/성공지표(초대율/공유율)를 고정.  
결과: 팀 초대율↑, 해지율↓.

**5) 안티패턴**  
- PRD가 “기능 설명서”만 되고 목표/지표가 없다   
- 너무 두꺼워 업데이트가 안 된다(죽은 문서)  
- 개발자 관점(구현)으로만 쓰고 사용자 관점(가치/행동)을 놓침 

**6) 체크리스트**  
- [ ] 목표/성공지표가 PRD 첫 페이지에 있다  
- [ ] In/Out 범위가 명시돼 있다  
- [ ] 이벤트/로그 설계가 있다  
- [ ] 오픈 이슈(Open Questions)가 있다

**7) 관련 지표/측정**  
- PRD에 포함된 성공지표(AARRR/퍼널 연결)  
- 출시 후: 전환율/리텐션/문의 감소 등

**8) 참고자료**  


#### PRD 샘플 템플릿(복붙용 + 일부 채움)

아래는 “문서 형태”를 익히기 위한 샘플입니다. (필드/목차를 그대로 쓰셔도 됩니다.)

- 기본 정보  
  - 기능명: (예) “오늘의 Top3 우선순위 추천”  
  - 목표 출시일: 2026-03-XX  
  - 담당: PM/디자인/개발  
  - 상태: Draft / In review / Approved

- 배경(Background)  
  - 관찰: 신규 유저 7일 리텐션 낮음  
  - 근거: (로그/UT/VOC 요약)

- 문제 정의(Problem)  
  - “신규 유저가 ‘내일 다시 올 이유’를 만들지 못한다”

- 목표/성공 지표(Goals & Metrics)  
  - 목표: 첫날 핵심 가치 경험을 30초 안에 제공  
  - 지표: Activation(Top3 설정 완료율), 7일 리텐션, 첫날 과업완료율

- 비목표(Non-goals)  
  - (예) 머신러닝 기반 개인화 모델 개발은 이번 범위 아님

- 타겟 사용자(Persona/Segment)  
  - (예) “계획이 자주 깨지는 직장인”

- 핵심 시나리오(Use Cases 요약)  
  - UC1: 신규 유저가 회원가입 후 Top3를 설정한다  
  - UC2: 유저가 추천된 Top3를 수정한다

- 요구사항(Functional Requirements)  
  - FR1: Top3 추천 리스트를 기본 제공  
  - FR2: 유저가 드래그로 순서를 바꿀 수 있음  
  - FR3: 설정 완료 시 ‘오늘 계획’ 화면으로 전환

- UX 산출물 링크  
  - User Flow / Wireframe / IA / Storyboard 링크(또는 첨부)

- 이벤트/로그 설계(Tracking Plan)  
  - event_top3_viewed, event_top3_confirmed, event_task_completed …

- 출시/롤아웃 계획  
  - 베타(10%) → 지표 확인 → 100%

- 리스크/가정/오픈 이슈  
  - 가정: Top3 추천이 ‘결정 피로’를 줄일 것이다  
  - 오픈: 추천 규칙(마감/반복/미완료)을 어떻게 가중치로 둘까?

### 개념 카드 D-03: IA(샘플 템플릿 포함)

**0) 한 줄 정의**  
IA는 “정보를 사람들이 찾기 쉽게, 구조로 설계하는 일”입니다. 

**1) 왜 필요한가**  
화면이 늘어날수록 사용자는 길을 잃습니다. IA는 메뉴/분류/라벨/탐색/검색을 묶어 “찾기·이해·탐색”을 가능하게 합니다. 

**2) 언제 쓰나**  
- 메뉴가 늘어나 UX가 복잡해질 때  
- B2B에서 권한/설정/관리 화면이 많을 때  
- 새 기능이 생길 때마다 “어디에 넣지?”가 문제일 때

**3) 어떻게 쓰나**  
- 사용자 목표별로 정보를 묶는다  
- 라벨을 사용자 언어로 맞춘다  
- 탐색(네비게이션)과 검색을 함께 설계한다  
- 최소 산출물: IA 트리 1장(Depth 2~3까지)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 기능이 늘어 홈이 복잡.  
PM의 선택: “오늘/모음/설정” 3개의 최상위로 재구조화.  
결과: 탐색 실패(되돌아가기)↓, 과업 성공률↑.

- 예시B(B2B)  
상황: 관리자 기능이 뒤죽박죽.  
PM의 선택: “관리(권한/감사로그)/분석/설정”으로 IA 재정렬.  
결과: CS 티켓↓, 활성 사용자↑.

**5) 안티패턴**  
- 조직 구조(팀/부서) 기준으로 메뉴를 만듦(사용자 목표와 불일치)  
- 라벨이 내부 용어(예: “프로덕트 오퍼레이션”)로 되어 있음  
- IA 없이 화면을 추가해 결국 리뉴얼 비용 폭발

**6) 체크리스트**  
- [ ] 최상위 메뉴는 사용자 목표로 설명된다  
- [ ] Depth가 깊어져도 사용자가 “지금 어디인지” 알 수 있다  
- [ ] 주요 페이지는 3클릭 이내 접근 가능(원칙, 맹신 금지)

**7) 지표/측정**  
- 탐색 성공률(UT), 검색 실패율, 페이지 이탈률, CS 티켓

**8) 참고자료**  
IA 정의(구조 설계/조직·라벨·검색·내비게이션의 종합)는  에서 정리됩니다. 

#### IA 샘플(트리)

B2C 투두 앱 IA(예)  
- 홈  
  - 오늘의 Top3  
  - 전체 할 일  
- 모음  
  - 프로젝트 목록  
  - 완료한 일  
- 설정  
  - 알림  
  - 계정/결제

B2B 대시보드 IA(예)  
- 대시보드  
  - 내 대시보드  
  - 템플릿  
- 공유  
  - 공유 링크  
  - 팀/권한  
  - 감사 로그  
- 분석  
  - 사용 현황  
  - 이벤트 설정  
- 설정  
  - 워크스페이스  
  - 결제/플랜

### 개념 카드 D-04: Use Case(샘플 템플릿 포함)

**0) 한 줄 정의**  
Use Case는 “사용자가 목표를 이루기 위해 시스템과 주고받는 행동 흐름”을 글로 적은 것입니다. 

**1) 왜 필요한가**  
Use Case는 “화면”이 아니라 “행동과 예외”를 다루어, 누락(예외 케이스)과 모호함을 줄입니다. 특히 개발/QA와의 합의에 강합니다. 

**2) 언제 쓰나**  
- 결제/권한/공유 등 예외가 많은 기능  
- B2B처럼 ‘역할/권한’이 복잡한 기능  
- QA 기준(수용 기준)이 필요한 기능

**3) 어떻게 쓰나**  
- Actor(주행위자)와 Goal을 정의  
- Preconditions(사전조건), Main Flow(기본 흐름), Alternate/Exception(대안/예외) 작성  
- Postconditions(사후조건)과 Acceptance Criteria(수용 기준)로 마무리  
- 최소 산출물: Main Flow + 2개 예외만이라도 작성

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 유저가 Top3를 저장한다.  
PM의 선택: “오프라인/네트워크 실패” 예외를 Use Case로 명시해 저장 UX를 설계.  
결과: 저장 실패 이슈↓, CS 문의↓.

- 예시B(B2B)  
상황: 공유 링크를 생성한다.  
PM의 선택: 권한(읽기/편집), 만료, 감사로그 기록을 Use Case로 명시.  
결과: 보안 이슈↓, 엔터프라이즈 전환↑.

**5) 안티패턴**  
- Use Case가 “기능 리스트”로 끝남(흐름/예외 없음)  
- Actor가 모호(“사용자”)만 적고 역할/권한이 없다  
- 수용 기준 없이 개발 완료 선언

**6) 체크리스트**  
- [ ] Goal이 한 문장으로 명확하다  
- [ ] 예외 흐름이 최소 2개 있다  
- [ ] 사후조건이 데이터 관점으로 적혀 있다(무엇이 저장/변경되나)

**7) 지표/측정**  
- 과업 성공률/실패율(UT/로그)  
- 예외 발생률(네트워크 오류, 권한 오류 등)

**8) 참고자료**  
Use Case의 고전적 정의는  의 설명(가치 있는 관찰 가능한 결과를 내는 행동 연쇄)에서 확인됩니다. 

#### Use Case 샘플 템플릿(채워진 예)

- Use Case ID: UC-SHARE-001  
- 제목: 공유 링크 생성  
- Primary Actor: 워크스페이스 관리자  
- Goal: 대시보드를 외부 협력사에게 읽기 전용으로 공유한다  
- Preconditions: 관리자는 로그인 상태, 대시보드가 존재  
- Trigger: “공유 링크 만들기” 버튼 클릭  
- Main Flow:  
  1. 시스템은 링크 설정 모달(권한/만료)을 보여준다  
   вместе… (생략 없이 실제로는 5~8단계로 작성)  
- Alternate Flow(예):  
  - A1: 만료일을 과거로 선택 → 에러 메시지  
- Exception(예):  
  - E1: 권한 없음 → 403/권한 안내  
- Postconditions:  
  - 링크가 저장되고, 감사 로그에 생성 이벤트가 기록됨  
- Acceptance Criteria:  
  - 만료된 링크 접근 시 404 또는 만료 안내 화면

### 개념 카드 D-07: Storyboard(샘플 템플릿 포함)

**0) 한 줄 정의**  
스토리보드는 “사용자가 제품을 쓰는 장면을 만화처럼 순서대로 그린 것”입니다. 

**1) 왜 필요한가**  
텍스트만으로는 팀이 같은 장면을 상상하기 어렵습니다. 스토리보드는 행동·감정·맥락을 한 번에 공유해, UX 의사결정 속도를 높입니다. 

**2) 언제 쓰나**  
- 온보딩/결제/공유처럼 감정이 중요한 여정  
- 이해관계자가 많아 “그림으로 설명”이 필요할 때  
- 기능이 아니라 “경험 가치”를 설득해야 할 때

**3) 어떻게 쓰나**  
- 페르소나와 목표를 먼저 정한다  
- 시작 장면(상황) → 문제 → 제품과의 상호작용 → 결과/감정 변화  
- 최소 산출물: 4~6컷(텍스트+간단 스케치)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 출근길, 해야 할 일이 많아 불안.  
PM의 선택: 스토리보드로 “30초 안에 Top3 결정” 장면을 설계해 온보딩 길이를 줄임.  
결과: 온보딩 완료율↑, 첫날 과업 완료율↑.

- 예시B(B2B)  
상황: 매주 보고서 제출 압박.  
PM의 선택: “템플릿 생성→공유 링크→상사 확인” 장면을 스토리보드로 설계.  
결과: 첫 공유까지 시간↓, 재방문↑.

**5) 안티패턴**  
- 기능 나열을 스토리보드라고 부름(장면/감정이 없음)  
- 스토리보드가 와이어프레임 대체로 변질(맥락이 사라짐)  
- 사용자 목표가 아니라 “우리 기능 자랑”이 목적이 됨

**6) 체크리스트**  
- [ ] 첫 컷에 “상황/문제”가 있다  
- [ ] 마지막 컷에 “감정 변화/성과”가 있다  
- [ ] 장면이 6컷 이하로 간결하다

**7) 지표/측정**  
- 스토리보드는 직접 지표가 아니라 “가설 명확성”을 높이는 도구  
- 후속 UT에서 해당 장면이 실제로 재현되는지 확인 

**8) 참고자료**

### 개념 카드 T-01: 프론트엔드/백엔드

**0) 한 줄 정의**  
프론트엔드는 “사용자 눈에 보이는 부분(화면)”이고, 백엔드는 “데이터/로직을 처리하는 뒤쪽(서버)”입니다. 

**1) 왜 필요한가**  
PM이 기술 구조를 모르면 “가능/불가능·비용·리스크” 감각 없이 요구를 던져 팀을 힘들게 합니다. 반대로 기본 구조를 알면 요구사항을 더 명확히, 더 현실적으로 전달할 수 있습니다. 

**2) 언제 쓰나**  
- “버튼 하나 추가”가 사실은 권한/DB 변경을 요구할 때  
- 성능/보안/로그 요구가 있을 때  
- API/데이터가 필요한 기능을 설계할 때

**3) 어떻게 쓰나**  
- 기능을 “화면(프론트)” + “데이터(백)” + “연결(API)”로 쪼개 설명한다  
- 최소 산출물: 요구사항에 “입력/처리/출력”을 명시(예: 입력값 검증, 에러 처리)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “Top3 추천” 버튼을 누르면 추천이 바로 뜨길 원함.  
PM의 선택: 프론트는 로딩/에러 상태를 정의, 백엔드는 추천 규칙/응답 시간을 정의.  
결과: 실패율↓, 사용자 불만↓.

- 예시B(B2B)  
상황: “공유 링크” 생성 시 만료/권한이 필요.  
PM의 선택: 백엔드에서 토큰 발급/만료, 프론트에서 안내/복사 UX 정의.  
결과: 보안 이슈↓, 공유율↑.

**5) 안티패턴**  
- “그냥 이렇게 해주세요”로 끝(데이터/예외 미정)  
- API 실패/지연 상황(로딩/재시도)을 정의하지 않음  
- “백엔드가 알아서”라고 넘김(결국 일정 폭탄)

**6) 체크리스트**  
- [ ] 성공 케이스뿐 아니라 실패/지연 케이스가 정의돼 있다  
- [ ] 권한/로그/보안이 필요한지 체크했다  
- [ ] 데이터가 어디서 오고 어디에 저장되는지 한 줄로 설명 가능

**7) 지표/측정**  
- 에러율, 응답시간, 재시도율, CS 문의

**8) 참고자료**  
클라이언트-서버 요청/응답 개념과 서버사이드 역할은  의 자료에서 쉽게 확인할 수 있습니다. 

### 개념 카드 T-02: UI 구성요소와 요구사항 전달 문장 예시(Header/GNB/LNB/FNB 등)

**0) 한 줄 정의**  
UI 구성요소는 “화면을 이루는 큰 블록(머리·몸통·발)”과 “이동 메뉴(전체/부분/하단)”입니다.

**1) 왜 필요한가**  
PM은 디자이너/개발자와 같은 단어로 대화할수록 오해가 줄고, 작업 범위가 명확해집니다.

**2) 언제 쓰나**  
- IA를 화면으로 풀어낼 때  
- 페이지 공통 요소(헤더/푸터/내비게이션)를 정의할 때  
- 스크롤/고정 요소(Sticky header 등) 요구가 있을 때

**3) 어떻게 쓰나(절차 + 최소 산출물)**  
- 공통 영역과 페이지별 가변 영역을 구분한다  
- 네비게이션(전역/로컬/하단)의 역할을 정의한다  
- 최소 산출물: 화면 와이어프레임 1장 + 공통 컴포넌트 정의 표

**4) 예시 2개(요구사항 문장 예시 포함)**  
- 예시A(B2C)  
상황: 홈에서 “오늘 Top3”를 보여주고, 하단 탭으로 이동.  
PM의 요구 문장 예시:  
  - “FNB는 ‘홈/모음/설정’ 3탭 고정이고, 홈 탭에서는 상단 Header를 스크롤해도 유지(Sticky)해주세요.”  
  - “Header의 오른쪽 아이콘은 ‘검색’이며, 탭 시 검색 화면으로 이동합니다.”

- 예시B(B2B)  
상황: 좌측 메뉴로 분석/설정 이동.  
PM의 요구 문장 예시:  
  - “GNB에는 워크스페이스 전환 드롭다운을 포함하고, LNB는 ‘대시보드/공유/분석/설정’ 4개 섹션을 보여주세요.”  
  - “권한이 없는 섹션은 LNB에서 숨기고(또는 비활성), 직접 URL 접근 시 권한 에러 페이지로 라우팅합니다.”

**5) 안티패턴**  
- “메뉴 좀 만들어주세요”처럼 구조/행동이 불명확  
- 고정요소(Sticky) 요구가 개발 난이도를 올리는데 정의 없이 요청  
- 반응형(모바일/웹)에서 GNB/FNB 룰이 없어서 깨짐

**6) 체크리스트**  
- [ ] 공통 요소(헤더/푸터/내비)와 페이지별 요소가 분리되어 있다  
- [ ] 고정/스크롤 행동이 정의돼 있다  
- [ ] 권한/로그인 상태별 메뉴 노출 규칙이 있다

**7) 지표/측정**  
- 탐색 성공률(UT), 메뉴 클릭 분포(히트맵), 이탈률

**8) 참고자료**  
(구성요소 자체는 업계 관용이지만, “사용자 행동 관찰/히트맵/퍼널”로 검증 가능) 

### 개념 카드 Q-01: 사용성 테스트(UT)

**0) 한 줄 정의**  
UT는 “사람에게 실제 과제를 시키고, 그 과정을 관찰해서 문제를 찾는 테스트”입니다. 

**1) 왜 필요한가**  
PM이 상상하는 UX와 사용자가 실제로 하는 행동은 다를 수 있습니다. UT는 ‘보이는 문제’를 빠르게 찾아 PRD를 현실로 업데이트하게 합니다. 

**2) 언제 쓰나**  
- 온보딩/결제/공유처럼 핵심 플로우  
- 출시 전(프로토타입) 또는 출시 후(개선)  
- 히트맵/로그에서 이상 신호가 보일 때

**3) 어떻게 쓰나(단계 + 최소 산출물)**  
- 과업(Task) 시나리오 작성(힌트를 주지 않기)   
- 참가자 모집(타겟 유저)  
- 진행(진행자/참가자/과업/관찰)   
- 분석(실패 지점/원인 가설/해결안)  
- 최소 산출물: UT 리포트 1장(Top5 이슈 + PRD 반영안)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: Top3 설정율이 낮음.  
PM의 선택: UT에서 과업 “오늘 할 일 3개를 설정해보세요” 진행 → 버튼 라벨이 오해를 부른다는 문제 발견.  
결과: 라벨/배치 개선 후 설정 완료율↑.

- 예시B(B2B)  
상황: 공유 링크 생성률 낮음.  
PM의 선택: UT에서 “읽기 전용 링크로 협력사에 공유하세요” 과업 → 권한 설명이 불명확해 불안으로 이탈하는 것 발견.  
결과: 안내/권한 기본값 개선 → 공유율↑.

**5) 안티패턴**  
- 과업이 “기능 설명”이 되어 힌트를 줌   
- 관찰 없이 “설문”으로 끝(행동을 못 봄)  
- 이슈를 나열하고 PRD/백로그로 연결하지 않음

**6) 체크리스트(복붙용)**  
- [ ] 과업 시나리오는 현실적이고, 해결 방법을 암시하지 않는다   
- [ ] 성공 기준(완료/시간/오류)을 정의했다  
- [ ] Top5 이슈에 대해 “원인 가설”과 “PRD 변경안”이 있다

**7) 관련 지표/측정**  
- 과업 성공률, 완료 시간, 오류/되돌리기 횟수  
- 정성: 혼잣말(think aloud)에서의 불만/혼란 표현 

**8) 참고자료**  


#### UT 테스트 과업 예시 10개(복붙용)

1. “회원가입 후 오늘 해야 할 일 3개를 추가하고 우선순위를 정해보세요.”  
2. “완료한 할 일을 ‘완료 목록’에서 찾아 다시 미완료로 바꿔보세요.”  
3. “알림 시간을 ‘내일 오전 9시’로 설정해보세요.”  
4. “지난주에 한 일을 요약해보세요(리포트 화면 찾기).”  
5. “팀원을 초대해 같은 대시보드를 보게 해보세요.”  
6. “읽기 전용 공유 링크를 만들고, 만료일을 7일 뒤로 설정해보세요.”  
7. “권한이 없는 화면에 들어가려고 할 때 무엇이 보이는지 설명해보세요.”  
8. “템플릿으로 새 대시보드를 만들고 첫 그래프를 추가해보세요.”  
9. “문제 상황(네트워크 끊김)에서 저장이 어떻게 되는지 확인해보세요.”  
10. “가격 플랜 비교 후 유료 전환을 시도해보세요(결제 직전까지).”

#### 관찰 포인트(실무)

- 사용자가 “멈추는” 순간(정지/머뭇거림)은 어디인가  
- 되돌리기/탭 전환/스크롤 반복은 무엇을 의미하는가  
- 용어(라벨) 이해 실패가 있는가  
- 에러/로딩 상태에서 불안이 커지는가  
- “목표 달성”이 아니라 “목표를 포기”하는 순간이 있는가

#### 분석 후 PRD 반영 방법(추천 포맷)

- 이슈(관찰) → 원인 가설 → 해결 옵션 A/B → 우선순위(임팩트×비용) → PRD 변경 문장(요구사항/수용 기준/추적 이벤트)

### 개념 카드 Q-02: 히트맵 분석



**0) 한 줄 정의**  
히트맵은 “사람들이 화면에서 어디를 많이 클릭/움직임/스크롤하는지 색으로 보여주는 지도”입니다. 

**1) 왜 필요한가**  
UT는 깊지만 표본이 작고, 로그는 넓지만 맥락이 약합니다. 히트맵은 “어디서 마찰이 생기는지”를 화면 단위로 빠르게 잡아, 우선순위를 정하는 데 유용합니다. 

**2) 언제 쓰나**  
- 랜딩 페이지 전환율 최적화  
- CTA 클릭이 기대와 다를 때  
- 긴 페이지에서 핵심 콘텐츠가 읽히는지 확인할 때

**3) 어떻게 쓰나**  
- 클릭/무브/스크롤 히트맵을 구분해 본다   
- “클릭되면 안 되는 곳”의 클릭(오해)을 찾는다  
- 스크롤 깊이로 CTA 위치를 검증한다  
- 최소 산출물: 히트맵 캡처 3장 + 인사이트 5개 + PRD 수정 항목

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: Top3 설정 버튼 클릭이 낮음.  
PM의 선택: 클릭 히트맵에서 사용자가 카드 상단 이미지를 클릭(클릭 가능한 줄 착각)하는 패턴 발견 → 버튼을 카드 전체 영역 클릭으로 확장.  
결과: 설정 시작률↑, Activation↑.

- 예시B(B2B)  
상황: 공유 링크 버튼이 잘 안 보임.  
PM의 선택: 스크롤 히트맵에서 60%가 버튼 이전에 이탈 → 버튼을 상단으로 이동.  
결과: 공유 클릭률↑.

**5) 안티패턴**  
- 히트맵만 보고 ‘왜’를 확정(인터뷰/UT로 보완해야 함)  
- 트래픽이 너무 적은데 결론(노이즈)  
- 세그먼트(모바일/데스크탑, 신규/기존) 없이 평균만 봄

**6) 체크리스트**  
- [ ] 히트맵은 세그먼트별로 본다  
- [ ] “기대 행동”과 “실제 행동” 차이를 문장으로 적었다  
- [ ] 인사이트가 PRD 변경 문장으로 연결됐다

**7) 지표/측정**  
- 클릭률, 스크롤 도달률, CTA 전환율  
- 보조: 세션 리플레이로 정성 맥락 보강 

**8) 참고자료**  
히트맵 유형(클릭/무브/스크롤 등)은 업계 도구 문서와 가이드에서 널리 정의됩니다. 

### 개념 카드 L-01: 레드 퀸 가설과 지속 개선

**0) 한 줄 정의**  
레드 퀸 가설은 “가만히 있어도 세상이 변하니, 살아남으려면 계속 달려야 한다”는 생각입니다. 

**1) 왜 필요한가**  
제품은 출시가 끝이 아니라 시작입니다. 경쟁/고객 기대/기술이 계속 변하므로, “측정→학습→개선” 루프를 조직 습관으로 만들어야 합니다. 레드 퀸 가설은 이 ‘지속 개선’의 사고틀로 자주 비유됩니다. 

**2) 언제 쓰나**  
- 출시 후 성과가 정체될 때  
- 경쟁사가 빠르게 따라올 때  
- 시장 규칙/플랫폼 정책이 바뀔 때

**3) 어떻게 쓰나**  
- 개선 루프를 루틴화: 주간 지표 리뷰 → 병목 가설 → 실험 → 회고  
- 최소 산출물: 개선 백로그 + 실험 로그(가설/결과)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 리텐션이 2개월째 정체.  
PM의 선택: 저니 맵+히트맵으로 새로운 마찰 지점 발굴, 2주마다 실험 1개 루틴화.  
결과: 리텐션이 점진적 상승.

- 예시B(B2B)  
상황: 경쟁사 신기능 출시로 이탈 증가.  
PM의 선택: “고객이 진짜로 이탈하는 이유”를 인터뷰로 재검증, 핵심 가치(공유/보안)를 강화하는 로드맵으로 재정렬.  
결과: churn 감소.

**5) 안티패턴**  
- 개선을 “큰 리뉴얼”로만 생각(작은 실험이 사라짐)  
- 지표 리뷰가 비난 회의가 됨(학습이 사라짐)  
- 경쟁 기능만 따라 하다 제품 정체성이 사라짐

**6) 체크리스트**  
- [ ] 실험이 ‘월 1회’가 아니라 ‘주 1회’(또는 격주)로 반복된다  
- [ ] 실패를 기록하고 학습으로 남긴다  
- [ ] 개선 우선순위가 지표/전략과 연결된다

**7) 지표/측정**  
- 실험 속도(월 실험 수), 학습 반영률, 장기 리텐션/NRR 변화

**8) 참고자료**  
레드 퀸 가설은 진화생물학에서 제안된 개념(1973)으로, “지속적 적응 필요”를 설명합니다. 

### 장 마무리: 실무 체크리스트와 안티패턴

실무 체크리스트  
- [ ] PRD는 “목적/성공지표가 먼저”다   
- [ ] UT/히트맵 결과는 반드시 “PRD 변경 문장”으로 되돌린다   
- [ ] 개선을 루틴화해 레드 퀸 상황(정체=후퇴)을 피한다   

자주 하는 실수(안티패턴)  
- 검증 없이 “개발 완료=성공” 선언  
- UT/히트맵 인사이트가 팀의 우선순위(백로그)에 반영되지 않음
