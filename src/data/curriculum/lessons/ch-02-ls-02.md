# Ch.2 — Lesson 2/7: 조직 구조와 채역 공고 역추적

> **학습 위치**: Chapter 2 · 2번째 레슨 / 전체 7개 레슨  
> **이전 단계**: 내가 성과를 낼 수 있는 조직 형태는 무엇일까요?  
> **다음 단계**: 이제 제품 설계의 핵심인 고객 이해 단계로 들어갑니다.

---

## 학습 목표
- 기능 조직 vs 목적 조직의 차이를 안다
- 채용 공고에서 요구 역량을 분석하는 법을 배운다

## 핵심 용어
- **기능 조직 vs 목적 조직**
- **채용 공고에서 요구하는 PM 역량**

---

**0) 한 줄 정의**  
기능 조직은 “기획/디자인/개발이 부서별로 릴레이”하고, 목적 조직은 “한 팀이 처음부터 끝까지 함께 책임”지는 방식입니다. 

**1) 왜 필요한가**  
조직 구조는 “문서 양”과 “회의 방식”을 결정합니다.  
기능 조직은 정렬을 위해 문서가 두꺼워지기 쉽고, 목적 조직은 빠른 실험을 위해 짧지만 빈번한 정렬이 중요합니다. (둘 다 장단점이 있습니다.) 

**2) 언제 쓰나**  
- “왜 이렇게 문서가 많아?” 혹은 “왜 이렇게 말로만 해?” 갈등이 생길 때  
- 요구사항 변경이 잦은데 Waterfall로 운영해 계속 ‘막판 폭발’이 날 때 

**3) 어떻게 쓰나**  
- 먼저 “변경 가능성(불확실성)”을 평가한다: 클수록 Agile/목적 조직의 이점이 커진다.   
- 기능 조직에서는 명시적 산출물(IA/Use Case/PRD)로 손실을 줄이고  
- 목적 조직에서는 “짧은 PRD + 프로토타입/UT”로 학습 속도를 높인다.

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 광고 캠페인에 맞춰 2주 안에 신규 온보딩 개선이 필요.  
PM의 선택: 크로스펑셔널 스쿼드로 2주 스프린트 구성(디자이너+개발자 상시).  
결과: 리드타임↓, Activation↑.

- 예시B(B2B)  
상황: 대기업 고객 커스터마이징 프로젝트, 요구사항이 많은 대신 변경은 상대적으로 적음.  
PM의 선택: 기능 조직+PJM을 세워 범위/납기/승인 프로세스를 견고히.  
결과: 납기 준수↑, 리워크↓.

**5) 안티패턴**  
- Waterfall을 쓰면서 “변경은 언제든 가능”하다고 약속(막판 폭발)   
- Agile을 쓰면서도 의사결정 권한이 분산(PO 부재, backlog 혼란)   
- 목적 조직인데 산출물 최소화가 아닌 “산출물 부재”로 흘러 커뮤니케이션 비용 증가

**6) 체크리스트**  
- [ ] 변경 가능성(불확실성) 수준에 맞춰 프로세스를 선택했다  
- [ ] 어떤 산출물은 “필수”이고(예: 성공지표/범위), 어떤 것은 “필요할 때만”인지 합의했다  
- [ ] 역할(결정권)과 루틴(주간/일간 정렬)을 명확히 했다

**7) 관련 지표**  
- 리드타임(아이디어→출시), 재작업률, 결함률, 고객 불만(VOC) 처리시간

**0) 한 줄 정의**  
채용 공고는 “회사들이 PM에게 실제로 돈 주고 사고 싶은 능력”의 목록입니다. 

![역량 지도 - 채용공고 분석](/generated/inline_ch6.svg) 

**1) 왜 필요한가**  
초심자는 “무엇부터 공부해야 할지” 막힙니다. 채용 공고의 “주요 업무/자격 요건”을 분해하면 PM 역량이 **문서·데이터·협업·도메인**으로 수렴합니다. 

**2) 언제 쓰나**  
- 커리어 전환/주니어 입문 시  
- 팀 내에서 “PM 역할 기대치”를 맞추고 싶을 때

**3) 어떻게 쓰나**  
- 공고 10개를 모아 공통 요구사항을 분류한다(문서화/데이터/커뮤니케이션/실행)  
- 최소 산출물: “역량 매트릭스(행=역량, 열=증빙 산출물)” 1장  
- 역량을 증빙할 산출물(PRD, IA, Use Case, UT 리포트 등)을 포트폴리오로 묶는다.

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “데이터 기반 개선”이 공통 요구인데 경험이 없다.  
PM의 선택: 실제 앱을 기준으로 AARRR 정의→이벤트 설계→가설 1개→간이 UT 5명→PRD 업데이트까지를 포트폴리오로 제작.  
결과: “데이터/실험/문서” 역량이 한 번에 증빙된다.

- 예시B(B2B)  
상황: “B2B 요구사항 정리”가 공고에서 반복.  
PM의 선택: Use Case + IA + 권한(roles) + 감사로그 요구사항까지 포함한 PRD 샘플 제작.  
결과: 현업형 문서화/리스크 인식이 드러난다.

**5) 안티패턴**  
- 공고의 “툴 이름”만 외우고, 실제 산출물/사고 과정이 없다  
- 포트폴리오가 UI만 예쁘고 “문제/지표/성과”가 비어 있다  
- “협업”을 말로만 쓰고, 실제 커뮤니케이션 산출물이 없다(회의록/결정 로그 등)

**6) 체크리스트**  
- [ ] 공고 10개 이상에서 반복되는 역량 5개를 뽑았다  
- [ ] 각 역량을 증빙할 산출물을 1개 이상 만들었다  
- [ ] 산출물에 “상황→선택→지표 변화” 이야기가 포함된다

**7) 지표/측정**  
- 포트폴리오 자체의 평가 기준: 문제 정의 명확성, 지표 적합성, 실행 가능성, 학습 루프 증빙
