## 고객 이해와 UX 설계

**0) 한 줄 정의**  
페르소나는 “우리 제품의 대표 사용자 한 명을 실제 사람처럼 구체적으로 그린 가짜 인물”입니다. 

**1) 왜 필요한가**  
“모든 사용자”를 만족시키려 하면 아무도 만족시키지 못합니다. 페르소나는 팀이 같은 사용자를 떠올리며 우선순위를 정하게 하고, 기능/문구/UX 결정의 기준이 됩니다. 

**2) 언제 쓰나**  
- 타겟이 넓어서 기능이 산으로 갈 때  
- 팀 내에서 “이 기능 누가 쓰는데?”가 반복될 때  
- 온보딩/핵심 플로우를 설계할 때

**3) 어떻게 쓰나(단계 + 최소 산출물)**  
- 리서치 기반 단서 모으기(인터뷰/설문/로그/VOC)  
- 사용자 군집(패턴) 2~3개로 묶기  
- 대표 페르소나 1~2개 작성(초심자는 1개부터)  
- 최소 산출물: 페르소나 카드 1장(목표/상황/니즈/페인포인트/대체재)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 투두 앱이 “학생/직장인/프리랜서” 모두를 겨냥해 기능이 복잡해짐.  
PM의 선택: “마감이 있는 업무를 놓치면 큰 손해”인 직장인을 대표 페르소나로 고정하고, 우선순위를 ‘마감 기반 정렬’로 통일.  
결과: 첫 주 과업 완료율↑, 리텐션↑.

- 예시B(B2B)  
상황: 대시보드 제품에 기능요청이 난립.  
PM의 선택: “보고서를 상사에게 매주 제출해야 하는 운영 매니저” 페르소나를 기준으로, ‘자동 리포트/공유’에 집중.  
결과: 활성 계정당 공유 횟수↑, 유료 전환↑.

**5) 안티패턴(3~5개)**  
- Bad: “20대 후반 스타트업 직원”처럼 너무 넓다(결정 기준이 안 된다).   
- Good처럼 보이지만 실제로는 ‘소설’: 리서치 근거가 없다  
- 페르소나가 “데모용 PPT”로 끝나고 backlog/PRD에 반영되지 않는다

**6) 체크리스트(복붙용)**  
- [ ] 페르소나에 “목표(Goal)”가 포함되어 있다  
- [ ] ‘대체재(지금 쓰는 방식/경쟁 서비스)’가 있다  
- [ ] 페인포인트가 3개 이하로 정리되어 있다  
- [ ] 기능 우선순위를 페르소나 기준으로 설명할 수 있다

**7) 관련 지표/측정**  
- 페르소나별 코호트 리텐션, 전환율 차이(가능하면 세그먼트로 본다)  
- 정성: 인터뷰에서 “이게 딱 내 문제”라는 발화 비율

Bad(나쁜 페르소나)  
- “20대 후반의 스타트업 직원”  
→ 너무 넓어 기능 결정이 안 된다(‘누구’인지가 아니라 ‘상황/목표/제약’이 비어 있음).   

Good(좋은 페르소나)  
- 이름: 정근(27)  
- 상황: 출퇴근 지하철에서 하루 계획을 빠르게 잡아야 함  
- 목표: “오늘 꼭 해야 하는 3개만 지키기”  
- 제약: 회사 일정이 자주 바뀌어 계획이 깨짐  
- 대체재: 메모앱 + 캘린더 + 머릿속 기억  
- 페인포인트: 우선순위가 계속 뒤집힘, 미완료가 쌓여 죄책감  
→ “어떤 UX가 도움이 되는지”가 바로 떠오른다(예: 3개 고정, 일정 변경 시 자동 재정렬).

**0) 한 줄 정의**  
데스크 리서치는 “사람을 만나기 전에, 이미 세상에 있는 자료로 시장과 사용자를 공부하는 것”입니다.

**1) 왜 필요한가**  
인터뷰는 비싸고 느립니다. 데스크 리서치로 기본 가설(누가/무엇을/어떤 대안으로 해결 중인지)을 세우면, 인터뷰 질문이 날카로워지고 실수(엉뚱한 시장 진입)가 줄어듭니다.

**2) 언제 쓰나**  
- 제품 아이디어 초기(Discover 단계)   
- 경쟁/대체재가 많은 시장에서 포지셔닝이 불명확할 때  
- 인터뷰 질문 설계 전

**3) 어떻게 쓰나**  
- 시장: 시장 규모/성장률/규제/핵심 플레이어  
- 사용자: 리뷰/커뮤니티/FAQ/검색어(문제 표현)  
- 제품: 경쟁사 온보딩/유료벽/핵심 가치 제안  
- 최소 산출물: “요약 1장 + 질문 10개” (인터뷰 가이드로 연결)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 투두 앱이 포화.  
PM의 선택: 앱스토어 리뷰에서 반복되는 불만(“우선순위가 어렵다”)을 추출해 ‘우선순위 자동화’가 틈새인지 검증.  
결과: 인터뷰 질문이 “기능 아이디어”가 아닌 “문제 강도/대체재” 중심으로 정교해짐.

- 예시B(B2B)  
상황: 대시보드 SaaS 경쟁이 치열.  
PM의 선택: 경쟁사 가격/권한/감사로그 유무를 비교표로 만들고, “보안/컴플라이언스”가 차별점이 될지 가설 수립.  
결과: 세일즈 메시지/PRD의 성공지표가 명확해짐.

**5) 안티패턴**  
- 자료를 많이 모았는데 결론이 없다(“그래서 우리가 뭘 할 건데?”)  
- “시장 규모”만 보고 사용자 문제를 안 본다  
- 경쟁사 기능을 복사하는 데스크 리서치(차별화가 사라짐)

**6) 체크리스트**  
- [ ] 사용자 문제를 ‘사용자 언어’로 10개 수집했다  
- [ ] 대체재(경쟁/비경쟁 포함)를 최소 5개 분석했다  
- [ ] 인터뷰에서 검증할 가설을 3개로 줄였다

**7) 지표/측정**  
- 데스크 리서치 자체는 지표가 아니라 “가설의 품질”로 평가  
- 후속 지표: 인터뷰에서 가설이 얼마나 빠르게 반박/강화되는지(learn cycle 속도)

(프로세스 맥락은 더블 다이아몬드의 Discover 단계와 연결)

**0) 한 줄 정의**  
고객 인터뷰는 “진짜 사용자가 왜 그렇게 행동하는지, 말로 들어보는 것”입니다.

![탐정의 책상 - 진짜 문제 찾기](/generated/detective_desk.svg)

**1) 왜 필요한가**  
로그는 “무슨 일이 일어났는지”는 알려주지만 “왜”는 끝까지 알려주지 않습니다. 인터뷰/IDI는 동기·맥락·대체재를 밝혀 PRD의 ‘왜’(문제 정의)를 단단하게 만듭니다. 

**2) 언제 쓰나**  
- 지표가 떨어졌는데 원인이 여러 개일 때  
- 새로운 가치 제안/가격/온보딩을 바꿀 때  
- VOC가 서로 충돌할 때(누구의 문제인지 분해 필요)

**3) 어떻게 쓰나**  
- 목적을 정한다: 탐색(문제 발견) vs 검증(가설 테스트)  
- 질문은 “기능” 대신 “상황/행동/대체재/결정 기준” 중심으로 만든다  
- 최소 산출물: 인터뷰 노트 → 인사이트 5개 → “PRD에 반영할 문장 3개”

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 유저가 알림을 꺼버림.  
PM의 선택: “왜 알림이 싫었나”를 상황 질문으로 파고들어(언제/무슨 이유로) 맞춤 알림 규칙을 설계.  
결과: 알림 차단율↓, 재방문↑.

- 예시B(B2B)  
상황: 팀원이 제품을 쓰지 않는다(시트로 회귀).  
PM의 선택: “업무 흐름 속 ‘인수인계’ 순간”에서 제품이 끊긴다는 걸 발견 후, 공유/권한/댓글 기능을 우선 개발.  
결과: 주간 활성 사용자(Team WAU)↑, 시트 의존↓.

**5) 안티패턴**  
- “이 기능 필요하세요?”처럼 유도 질문  
- 사용자가 말한 해결책을 그대로 구현(문제와 해결책을 혼동)  
- 인터뷰 후 “인사이트”가 아니라 “감상문”만 남는다

**6) 체크리스트**  
- [ ] 인터뷰 목적(탐색/검증)을 한 문장으로 썼다  
- [ ] 대체재(현재 어떻게 해결하는지)를 반드시 물었다  
- [ ] “결정 기준”을 물었다(왜 A를 선택하고 B를 포기했나)  
- [ ] 인사이트를 PRD 문장으로 바꿔 적었다

**7) 지표/측정**  
- 인터뷰 품질: 재현 가능한 패턴(반복되는 상황/행동)이 나오는지  
- 후속 지표: 개선 후 과업 성공률, 리텐션 변화

사용성 테스트와의 차이를 포함해 UX 리서치의 주요 방법으로 널리 설명됨.

**0) 한 줄 정의**  
VOC는 “고객이 남긴 불만/요청/칭찬을 모아, 제품 개선 순서를 정하는 재료”입니다.

**1) 왜 필요한가**  
VOC를 ‘좋은 말/나쁜 말’로만 보면, 제품이 “소음에 끌려다니는” 상태가 됩니다. VOC를 분류·정량화·우선순위화해야 PM이 ‘학습’으로 전환할 수 있습니다.

**2) 언제 쓰나**  
- 고객문의/리뷰가 늘어나는데 해결이 반복될 때  
- 리텐션/전환이 떨어지는데 행동 로그만으로 원인이 모호할 때

**3) 어떻게 쓰나**  
- 카테고리(온보딩/결제/성능/콘텐츠/권한 등)로 태깅  
- 심각도(빈도×임팩트×대체 가능성)로 우선순위  
- 최소 산출물: VOC Top10 + “PRD 반영 여부/시점” 표

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “우선순위가 헷갈려요” VOC 급증.  
PM의 선택: VOC를 ‘결정 피로’ 카테고리로 묶고, 첫 화면에서 3개 추천+수정 방식으로 단순화.  
결과: 첫 주 이탈↓, 리뷰 평점↑.

- 예시B(B2B)  
상황: “누가 뭘 봤는지 모르겠다” VOC 반복.  
PM의 선택: 감사로그/권한 요구를 한 덩어리로 정의하고, 엔터프라이즈 플랜의 핵심 가치로 PRD에 반영.  
결과: 업셀 전환↑, 이탈(해지) 감소.

**5) 안티패턴**  
- VOC 수집만 하고 제품 의사결정으로 연결되지 않는다  
- “큰 고객” 의견만 반영해 제품이 왜곡된다  
- VOC를 기능 리스트로만 취급(근본 원인을 놓침)

**6) 체크리스트**  
- [ ] VOC는 태그/카테고리로 분류된다  
- [ ] 우선순위 기준이 지표(리텐션/전환/매출)와 연결된다  
- [ ] VOC→가설→실험→결과의 추적이 가능하다

**7) 지표/측정**  
- VOC 처리시간, 동일 이슈 재발률, CS 티켓 감소율  
- 제품 지표: 리텐션/전환 개선으로 연결되는지

(원문 기반 운영 개념 확장)

**0) 한 줄 정의**  
UT는 “사람에게 실제 과제를 시키고, 그 과정을 관찰해서 문제를 찾는 테스트”입니다. 

**1) 왜 필요한가**  
PM이 상상하는 UX와 사용자가 실제로 하는 행동은 다를 수 있습니다. UT는 ‘보이는 문제’를 빠르게 찾아 PRD를 현실로 업데이트하게 합니다. 

**2) 언제 쓰나**  
- 온보딩/결제/공유처럼 핵심 플로우  
- 출시 전(프로토타입) 또는 출시 후(개선)  
- 히트맵/로그에서 이상 신호가 보일 때

**3) 어떻게 쓰나(단계 + 최소 산출물)**  
- 과업(Task) 시나리오 작성(힌트를 주지 않기)   
- 참가자 모집(타겟 유저)  
- 진행(진행자/참가자/과업/관찰)   
- 분석(실패 지점/원인 가설/해결안)  
- 최소 산출물: UT 리포트 1장(Top5 이슈 + PRD 반영안)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: Top3 설정율이 낮음.  
PM의 선택: UT에서 과업 “오늘 할 일 3개를 설정해보세요” 진행 → 버튼 라벨이 오해를 부른다는 문제 발견.  
결과: 라벨/배치 개선 후 설정 완료율↑.

- 예시B(B2B)  
상황: 공유 링크 생성률 낮음.  
PM의 선택: UT에서 “읽기 전용 링크로 협력사에 공유하세요” 과업 → 권한 설명이 불명확해 불안으로 이탈하는 것 발견.  
결과: 안내/권한 기본값 개선 → 공유율↑.

**5) 안티패턴**  
- 과업이 “기능 설명”이 되어 힌트를 줌   
- 관찰 없이 “설문”으로 끝(행동을 못 봄)  
- 이슈를 나열하고 PRD/백로그로 연결하지 않음

**6) 체크리스트(복붙용)**  
- [ ] 과업 시나리오는 현실적이고, 해결 방법을 암시하지 않는다   
- [ ] 성공 기준(완료/시간/오류)을 정의했다  
- [ ] Top5 이슈에 대해 “원인 가설”과 “PRD 변경안”이 있다

**7) 관련 지표/측정**  
- 과업 성공률, 완료 시간, 오류/되돌리기 횟수  
- 정성: 혼잣말(think aloud)에서의 불만/혼란 표현 

1. “회원가입 후 오늘 해야 할 일 3개를 추가하고 우선순위를 정해보세요.”  
2. “완료한 할 일을 ‘완료 목록’에서 찾아 다시 미완료로 바꿔보세요.”  
3. “알림 시간을 ‘내일 오전 9시’로 설정해보세요.”  
4. “지난주에 한 일을 요약해보세요(리포트 화면 찾기).”  
5. “팀원을 초대해 같은 대시보드를 보게 해보세요.”  
6. “읽기 전용 공유 링크를 만들고, 만료일을 7일 뒤로 설정해보세요.”  
7. “권한이 없는 화면에 들어가려고 할 때 무엇이 보이는지 설명해보세요.”  
8. “템플릿으로 새 대시보드를 만들고 첫 그래프를 추가해보세요.”  
9. “문제 상황(네트워크 끊김)에서 저장이 어떻게 되는지 확인해보세요.”  
10. “가격 플랜 비교 후 유료 전환을 시도해보세요(결제 직전까지).”

- 사용자가 “멈추는” 순간(정지/머뭇거림)은 어디인가  
- 되돌리기/탭 전환/스크롤 반복은 무엇을 의미하는가  
- 용어(라벨) 이해 실패가 있는가  
- 에러/로딩 상태에서 불안이 커지는가  
- “목표 달성”이 아니라 “목표를 포기”하는 순간이 있는가

- 이슈(관찰) → 원인 가설 → 해결 옵션 A/B → 우선순위(임팩트×비용) → PRD 변경 문장(요구사항/수용 기준/추적 이벤트)
