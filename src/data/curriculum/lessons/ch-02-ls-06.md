## 구현과 운영

**0) 한 줄 정의**  
더블 다이아몬드는 “문제를 넓게 찾고(확산), 핵심을 정한 뒤(수렴), 해결책을 넓게 만들고(확산), 최종안을 정하는(수렴) 작업 흐름”입니다. 

**1) 왜 필요한가**  
초심자의 흔한 실수는 “바로 해결책부터 만든다”입니다. 더블 다이아몬드는 그 반대로, **문제 정의의 품질**이 결과를 좌우한다는 점을 구조로 강제합니다. 

**2) 언제 쓰나**  
- 신규 기능/신사업처럼 불확실성이 큰 작업  
- UX/전환 문제가 복합적일 때  
- 팀이 “문제”가 아니라 “아이디어”로 싸울 때

**3) 어떻게 쓰나(절차 + 최소 산출물)**  
- Discover: 조사/인터뷰/데스크 리서치로 문제 공간 확장  
- Define: 페르소나/저니 맵으로 핵심 문제 정의  
- Develop: 아이디어/프로토타입/MVP 설계  
- Deliver: UT/데이터로 검증 후 출시(또는 반복)   
최소 산출물은 “문제 정의 1장 + MVP 가설 1장 + 검증 리포트 1장”입니다.

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 유입은 많은데 재방문이 낮다(리텐션 문제).  
PM의 선택: Discover/Define에서 “우선순위 결정 피로”가 핵심임을 찾고, Develop에서 “오늘 3개 추천” MVP를 만들고 UT로 검증 후 Deliver.  
결과: 7일 리텐션↑.

- 예시B(B2B)  
상황: 도입은 되는데 확산이 안 된다.  
PM의 선택: Define에서 “팀이 함께 보는 순간(공유/권한)”이 가치임을 정의, Develop에서 공유 플로우 MVP 제작, Deliver에서 세그먼트별 전환 측정.  
결과: 팀 초대율↑, 해지율↓.

**5) 안티패턴**  
- Define 없이 바로 Develop(해결책 도박)  
- Discover를 “자료 모으기”로 끝내고 통찰이 없다  
- Deliver에서 측정 없이 ‘감’으로 성공 선언

**6) 체크리스트**  
- [ ] Define 단계에서 “핵심 문제 문장”이 있다  
- [ ] MVP는 학습 목표(검증할 가설)가 명확하다   
- [ ] Deliver에서 성공 기준(지표)이 고정돼 있다

**7) 지표/측정**  
- Define 품질: UT에서 과업 실패 원인이 문제 정의와 맞는지  
- Deliver 결과: AARRR 해당 단계의 지표 변화

더블 다이아몬드는  에 의해 널리 소개된 디자인/혁신 프로세스입니다.

| 기간 | 목표 | Discover 산출물 | Define 산출물 | Develop 산출물 | Deliver 산출물 |
|---|---|---|---|---|---|
| 2주 | 빠른 개선 실험 | VOC/로그 요약 1장 | 핵심 문제 1문장 + 성공지표 | 저충실도 프로토타입 | UT 5명 + 지표 1주 측정 |
| 4주 | 기능 MVP 출시 | 인터뷰 6~8명, 경쟁 분석 | 페르소나 1개 + 저니 맵 | MVP 범위(In/Out) + PRD 초안 | 베타 출시 + 코호트 리텐션 측정 |
| 8주 | 신규 가치 제안 검증 | 시장/세그먼트 조사 | 문제-가치 가설, 포지셔닝 | MVP 2안 비교(A/B) | 정식 출시 판단(Go/No-Go) |

**0) 한 줄 정의**  
MVP는 “최소한으로 만들고, 최대한 빨리 배우기 위한 첫 버전”입니다. 

> [!TIP]
> **MVP에 대한 깊이 있는 실전 프레임워크(MLP, PoC, V1 구분 등)는 [Chapter 4 - Lesson 4]에서 상세하게 다룹니다.** 시니어 PM의 MVP 설계 비법이 궁금하다면 해당 레슨을 먼저 확인해 보세요!

**1) 왜 필요한가**  
불확실한 시장에서 ‘완벽한 제품’은 늦고 비쌉니다. MVP는 최소 노력으로 검증 학습을 얻어, 잘못된 방향에 큰 비용을 쓰는 위험을 줄입니다. 

**2) 언제 쓰나**  
- 우리가 풀고자 하는 문제가 “진짜로 존재하는지”가 불확실할 때  
- 특정 기능이 리텐션/매출에 영향이 있을지 모르겠을 때  
- 가격/패키징 같은 민감한 변경 전

**3) 어떻게 쓰나**  
- “검증 가설”을 하나로 고정한다  
- 그 가설을 검증하는 데 필요한 최소 기능만 만든다  
- 측정 이벤트/성공 기준을 먼저 정의한다  
- 최소 산출물: 가설 1문장 + MVP 범위 표 + 측정 계획

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “우선순위 추천”이 리텐션을 올릴지 모름.  
PM의 선택: ML 모델을 만들기 전에 규칙 기반(마감/반복/미완료) 추천으로 MVP 구현.  
결과: 추천 사용률↑, 리텐션 변화로 가설 일부 검증.

- 예시B(B2B)  
상황: “공유 링크”가 구독 전환에 영향?  
PM의 선택: 완전한 권한 시스템 전, 읽기 전용 공유 링크 + 만료 옵션만 MVP로 출시.  
결과: 공유율↑, 계약 전환 퍼널에서 병목 완화.

**5) 안티패턴**  
- MVP를 “저품질 제품”로 오해(최소는 ‘학습’ 기준이지 ‘대충’이 아님)   
- 가설/성공 기준 없이 MVP를 만듦  
- “기술 부채”를 방치해 다음 반복이 느려짐

**6) 체크리스트**  
- [ ] 가설이 “측정 가능한 문장”이다  
- [ ] 만들지 않을 것(Out of scope)을 명시했다  
- [ ] 성공 기준을 지표로 적었다

**7) 지표/측정**  
- Activation/Retention의 변화(가설에 따라)  
- Time-to-Value 변화(첫 가치 도달 시간)

MVP 정의는  의 설명(검증 학습 최대화/최소 노력)에서 널리 인용됩니다.

**0) 한 줄 정의**  
PRD는 “만들 제품(기능)이 무엇이고, 왜 만들며, 성공을 어떻게 볼지”를 팀이 합의하는 문서입니다. 

**1) 왜 필요한가**  
PRD는 ‘상세 스펙’이라기보다 **정렬(alignment)** 입니다. 목적/성공기준이 없으면 개발은 진행돼도 “원하는 결과”가 나지 않습니다. 

**2) 언제 쓰나**  
- 개발 착수 전에 “왜/무엇/성공”을 합의해야 할 때  
- 이해관계자(영업/CS/경영진)가 많을 때  
- 여러 팀이 동시에 같은 기능을 만질 때

**3) 어떻게 쓰나(단계 + 최소 산출물)**  
- 문제/목표/성공지표를 먼저 쓰고  
- 사용자/시나리오(Use Case)로 요구를 구체화하고  
- 범위(In/Out)와 리스크를 명시하고  
- 추적(이벤트/로그)과 출시 계획을 넣습니다. 

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 리텐션 낮음.  
PM의 선택: “첫날 3개 완료”를 목표로 PRD 작성 → 온보딩/추천/리마인드 범위를 정함.  
결과: Activation↑, 리텐션↑.

- 예시B(B2B)  
상황: 팀 확산 부족.  
PM의 선택: “공유/권한” 기능 PRD로 범위/성공지표(초대율/공유율)를 고정.  
결과: 팀 초대율↑, 해지율↓.

**5) 안티패턴**  
- PRD가 “기능 설명서”만 되고 목표/지표가 없다   
- 너무 두꺼워 업데이트가 안 된다(죽은 문서)  
- 개발자 관점(구현)으로만 쓰고 사용자 관점(가치/행동)을 놓침 

**6) 체크리스트**  
- [ ] 목표/성공지표가 PRD 첫 페이지에 있다  
- [ ] In/Out 범위가 명시돼 있다  
- [ ] 이벤트/로그 설계가 있다  
- [ ] 오픈 이슈(Open Questions)가 있다

**7) 관련 지표/측정**  
- PRD에 포함된 성공지표(AARRR/퍼널 연결)  
- 출시 후: 전환율/리텐션/문의 감소 등

아래는 “문서 형태”를 익히기 위한 샘플입니다. (필드/목차를 그대로 쓰셔도 됩니다.)

- 기본 정보  
  - 기능명: (예) “오늘의 Top3 우선순위 추천”  
  - 목표 출시일: 2026-03-XX  
  - 담당: PM/디자인/개발  
  - 상태: Draft / In review / Approved

- 배경(Background)  
  - 관찰: 신규 유저 7일 리텐션 낮음  
  - 근거: (로그/UT/VOC 요약)

- 문제 정의(Problem)  
  - “신규 유저가 ‘내일 다시 올 이유’를 만들지 못한다”

- 목표/성공 지표(Goals & Metrics)  
  - 목표: 첫날 핵심 가치 경험을 30초 안에 제공  
  - 지표: Activation(Top3 설정 완료율), 7일 리텐션, 첫날 과업완료율

- 비목표(Non-goals)  
  - (예) 머신러닝 기반 개인화 모델 개발은 이번 범위 아님

- 타겟 사용자(Persona/Segment)  
  - (예) “계획이 자주 깨지는 직장인”

- 핵심 시나리오(Use Cases 요약)  
  - UC1: 신규 유저가 회원가입 후 Top3를 설정한다  
  - UC2: 유저가 추천된 Top3를 수정한다

- 요구사항(Functional Requirements)  
  - FR1: Top3 추천 리스트를 기본 제공  
  - FR2: 유저가 드래그로 순서를 바꿀 수 있음  
  - FR3: 설정 완료 시 ‘오늘 계획’ 화면으로 전환

- UX 산출물 링크  
  - User Flow / Wireframe / IA / Storyboard 링크(또는 첨부)

- 이벤트/로그 설계(Tracking Plan)  
  - event_top3_viewed, event_top3_confirmed, event_task_completed …

- 출시/롤아웃 계획  
  - 베타(10%) → 지표 확인 → 100%

- 리스크/가정/오픈 이슈  
  - 가정: Top3 추천이 ‘결정 피로’를 줄일 것이다  
  - 오픈: 추천 규칙(마감/반복/미완료)을 어떻게 가중치로 둘까?

**0) 한 줄 정의**  
IA는 “정보를 사람들이 찾기 쉽게, 구조로 설계하는 일”입니다. 

**1) 왜 필요한가**  
화면이 늘어날수록 사용자는 길을 잃습니다. IA는 메뉴/분류/라벨/탐색/검색을 묶어 “찾기·이해·탐색”을 가능하게 합니다. 

**2) 언제 쓰나**  
- 메뉴가 늘어나 UX가 복잡해질 때  
- B2B에서 권한/설정/관리 화면이 많을 때  
- 새 기능이 생길 때마다 “어디에 넣지?”가 문제일 때

**3) 어떻게 쓰나**  
- 사용자 목표별로 정보를 묶는다  
- 라벨을 사용자 언어로 맞춘다  
- 탐색(네비게이션)과 검색을 함께 설계한다  
- 최소 산출물: IA 트리 1장(Depth 2~3까지)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 기능이 늘어 홈이 복잡.  
PM의 선택: “오늘/모음/설정” 3개의 최상위로 재구조화.  
결과: 탐색 실패(되돌아가기)↓, 과업 성공률↑.

- 예시B(B2B)  
상황: 관리자 기능이 뒤죽박죽.  
PM의 선택: “관리(권한/감사로그)/분석/설정”으로 IA 재정렬.  
결과: CS 티켓↓, 활성 사용자↑.

**5) 안티패턴**  
- 조직 구조(팀/부서) 기준으로 메뉴를 만듦(사용자 목표와 불일치)  
- 라벨이 내부 용어(예: “프로덕트 오퍼레이션”)로 되어 있음  
- IA 없이 화면을 추가해 결국 리뉴얼 비용 폭발

**6) 체크리스트**  
- [ ] 최상위 메뉴는 사용자 목표로 설명된다  
- [ ] Depth가 깊어져도 사용자가 “지금 어디인지” 알 수 있다  
- [ ] 주요 페이지는 3클릭 이내 접근 가능(원칙, 맹신 금지)

**7) 지표/측정**  
- 탐색 성공률(UT), 검색 실패율, 페이지 이탈률, CS 티켓

IA 정의(구조 설계/조직·라벨·검색·내비게이션의 종합)는  에서 정리됩니다.

B2C 투두 앱 IA(예)  
- 홈  
  - 오늘의 Top3  
  - 전체 할 일  
- 모음  
  - 프로젝트 목록  
  - 완료한 일  
- 설정  
  - 알림  
  - 계정/결제

B2B 대시보드 IA(예)  
- 대시보드  
  - 내 대시보드  
  - 템플릿  
- 공유  
  - 공유 링크  
  - 팀/권한  
  - 감사 로그  
- 분석  
  - 사용 현황  
  - 이벤트 설정  
- 설정  
  - 워크스페이스  
  - 결제/플랜

**0) 한 줄 정의**  
Use Case는 “사용자가 목표를 이루기 위해 시스템과 주고받는 행동 흐름”을 글로 적은 것입니다. 

**1) 왜 필요한가**  
Use Case는 “화면”이 아니라 “행동과 예외”를 다루어, 누락(예외 케이스)과 모호함을 줄입니다. 특히 개발/QA와의 합의에 강합니다. 

**2) 언제 쓰나**  
- 결제/권한/공유 등 예외가 많은 기능  
- B2B처럼 ‘역할/권한’이 복잡한 기능  
- QA 기준(수용 기준)이 필요한 기능

**3) 어떻게 쓰나**  
- Actor(주행위자)와 Goal을 정의  
- Preconditions(사전조건), Main Flow(기본 흐름), Alternate/Exception(대안/예외) 작성  
- Postconditions(사후조건)과 Acceptance Criteria(수용 기준)로 마무리  
- 최소 산출물: Main Flow + 2개 예외만이라도 작성

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 유저가 Top3를 저장한다.  
PM의 선택: “오프라인/네트워크 실패” 예외를 Use Case로 명시해 저장 UX를 설계.  
결과: 저장 실패 이슈↓, CS 문의↓.

- 예시B(B2B)  
상황: 공유 링크를 생성한다.  
PM의 선택: 권한(읽기/편집), 만료, 감사로그 기록을 Use Case로 명시.  
결과: 보안 이슈↓, 엔터프라이즈 전환↑.

**5) 안티패턴**  
- Use Case가 “기능 리스트”로 끝남(흐름/예외 없음)  
- Actor가 모호(“사용자”)만 적고 역할/권한이 없다  
- 수용 기준 없이 개발 완료 선언

**6) 체크리스트**  
- [ ] Goal이 한 문장으로 명확하다  
- [ ] 예외 흐름이 최소 2개 있다  
- [ ] 사후조건이 데이터 관점으로 적혀 있다(무엇이 저장/변경되나)

**7) 지표/측정**  
- 과업 성공률/실패율(UT/로그)  
- 예외 발생률(네트워크 오류, 권한 오류 등)

Use Case의 고전적 정의는  의 설명(가치 있는 관찰 가능한 결과를 내는 행동 연쇄)에서 확인됩니다.

- Use Case ID: UC-SHARE-001  
- 제목: 공유 링크 생성  
- Primary Actor: 워크스페이스 관리자  
- Goal: 대시보드를 외부 협력사에게 읽기 전용으로 공유한다  
- Preconditions: 관리자는 로그인 상태, 대시보드가 존재  
- Trigger: “공유 링크 만들기” 버튼 클릭  
- Main Flow:  
  1. 시스템은 링크 설정 모달(권한/만료)을 보여준다  
   вместе… (생략 없이 실제로는 5~8단계로 작성)  
- Alternate Flow(예):  
  - A1: 만료일을 과거로 선택 → 에러 메시지  
- Exception(예):  
  - E1: 권한 없음 → 403/권한 안내  
- Postconditions:  
  - 링크가 저장되고, 감사 로그에 생성 이벤트가 기록됨  
- Acceptance Criteria:  
  - 만료된 링크 접근 시 404 또는 만료 안내 화면

**0) 한 줄 정의**  
스토리보드는 “사용자가 제품을 쓰는 장면을 만화처럼 순서대로 그린 것”입니다. 

**1) 왜 필요한가**  
텍스트만으로는 팀이 같은 장면을 상상하기 어렵습니다. 스토리보드는 행동·감정·맥락을 한 번에 공유해, UX 의사결정 속도를 높입니다. 

**2) 언제 쓰나**  
- 온보딩/결제/공유처럼 감정이 중요한 여정  
- 이해관계자가 많아 “그림으로 설명”이 필요할 때  
- 기능이 아니라 “경험 가치”를 설득해야 할 때

**3) 어떻게 쓰나**  
- 페르소나와 목표를 먼저 정한다  
- 시작 장면(상황) → 문제 → 제품과의 상호작용 → 결과/감정 변화  
- 최소 산출물: 4~6컷(텍스트+간단 스케치)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: 출근길, 해야 할 일이 많아 불안.  
PM의 선택: 스토리보드로 “30초 안에 Top3 결정” 장면을 설계해 온보딩 길이를 줄임.  
결과: 온보딩 완료율↑, 첫날 과업 완료율↑.

- 예시B(B2B)  
상황: 매주 보고서 제출 압박.  
PM의 선택: “템플릿 생성→공유 링크→상사 확인” 장면을 스토리보드로 설계.  
결과: 첫 공유까지 시간↓, 재방문↑.

**5) 안티패턴**  
- 기능 나열을 스토리보드라고 부름(장면/감정이 없음)  
- 스토리보드가 와이어프레임 대체로 변질(맥락이 사라짐)  
- 사용자 목표가 아니라 “우리 기능 자랑”이 목적이 됨

**6) 체크리스트**  
- [ ] 첫 컷에 “상황/문제”가 있다  
- [ ] 마지막 컷에 “감정 변화/성과”가 있다  
- [ ] 장면이 6컷 이하로 간결하다

**7) 지표/측정**  
- 스토리보드는 직접 지표가 아니라 “가설 명확성”을 높이는 도구  
- 후속 UT에서 해당 장면이 실제로 재현되는지 확인 

| 컷 | 장면 | 사용자 감정 | PM이 확인해야 할 포인트 |
|---|---|---|---|
| 1 | 출근길, 알림이 쏟아짐 | 😫 | 문제 강도(스트레스) |
| 2 | 메모앱/캘린더를 오가며 혼란 | 😵 | 대체재의 불편 |
| 3 | 앱 설치, “오늘 Top3” 안내 | 🙂 기대 | 첫 가치 제안 |
| 4 | 30초 안에 3개 선택 완료 | 🙂 안정 | 입력 마찰 |
| 5 | 퇴근 전 완료 체크, 작은 성취 | 🙂 만족 | 피드백 설계 |
| 6 | 다음날 자동 리마인드로 재방문 | 🙂 습관 | 리텐션 루프 |

**0) 한 줄 정의**  
프론트엔드는 “사용자 눈에 보이는 부분(화면)”이고, 백엔드는 “데이터/로직을 처리하는 뒤쪽(서버)”입니다.

![정밀 기어 - IT 아키텍처와 협업](/generated/precision_gears.svg) 

**1) 왜 필요한가**  
PM이 기술 구조를 모르면 “가능/불가능·비용·리스크” 감각 없이 요구를 던져 팀을 힘들게 합니다. 반대로 기본 구조를 알면 요구사항을 더 명확히, 더 현실적으로 전달할 수 있습니다. 

**2) 언제 쓰나**  
- “버튼 하나 추가”가 사실은 권한/DB 변경을 요구할 때  
- 성능/보안/로그 요구가 있을 때  
- API/데이터가 필요한 기능을 설계할 때

**3) 어떻게 쓰나**  
- 기능을 “화면(프론트)” + “데이터(백)” + “연결(API)”로 쪼개 설명한다  
- 최소 산출물: 요구사항에 “입력/처리/출력”을 명시(예: 입력값 검증, 에러 처리)

**4) 예시 2개(가상)**  
- 예시A(B2C)  
상황: “Top3 추천” 버튼을 누르면 추천이 바로 뜨길 원함.  
PM의 선택: 프론트는 로딩/에러 상태를 정의, 백엔드는 추천 규칙/응답 시간을 정의.  
결과: 실패율↓, 사용자 불만↓.

- 예시B(B2B)  
상황: “공유 링크” 생성 시 만료/권한이 필요.  
PM의 선택: 백엔드에서 토큰 발급/만료, 프론트에서 안내/복사 UX 정의.  
결과: 보안 이슈↓, 공유율↑.

**5) 안티패턴**  
- “그냥 이렇게 해주세요”로 끝(데이터/예외 미정)  
- API 실패/지연 상황(로딩/재시도)을 정의하지 않음  
- “백엔드가 알아서”라고 넘김(결국 일정 폭탄)

**6) 체크리스트**  
- [ ] 성공 케이스뿐 아니라 실패/지연 케이스가 정의돼 있다  
- [ ] 권한/로그/보안이 필요한지 체크했다  
- [ ] 데이터가 어디서 오고 어디에 저장되는지 한 줄로 설명 가능

**7) 지표/측정**  
- 에러율, 응답시간, 재시도율, CS 문의

클라이언트-서버 요청/응답 개념과 서버사이드 역할은  의 자료에서 쉽게 확인할 수 있습니다.

**0) 한 줄 정의**  
UI 구성요소는 “화면을 이루는 큰 블록(머리·몸통·발)”과 “이동 메뉴(전체/부분/하단)”입니다.

**1) 왜 필요한가**  
PM은 디자이너/개발자와 같은 단어로 대화할수록 오해가 줄고, 작업 범위가 명확해집니다.

**2) 언제 쓰나**  
- IA를 화면으로 풀어낼 때  
- 페이지 공통 요소(헤더/푸터/내비게이션)를 정의할 때  
- 스크롤/고정 요소(Sticky header 등) 요구가 있을 때

**3) 어떻게 쓰나(절차 + 최소 산출물)**  
- 공통 영역과 페이지별 가변 영역을 구분한다  
- 네비게이션(전역/로컬/하단)의 역할을 정의한다  
- 최소 산출물: 화면 와이어프레임 1장 + 공통 컴포넌트 정의 표

**4) 예시 2개(요구사항 문장 예시 포함)**  
- 예시A(B2C)  
상황: 홈에서 “오늘 Top3”를 보여주고, 하단 탭으로 이동.  
PM의 요구 문장 예시:  
  - “FNB는 ‘홈/모음/설정’ 3탭 고정이고, 홈 탭에서는 상단 Header를 스크롤해도 유지(Sticky)해주세요.”  
  - “Header의 오른쪽 아이콘은 ‘검색’이며, 탭 시 검색 화면으로 이동합니다.”

- 예시B(B2B)  
상황: 좌측 메뉴로 분석/설정 이동.  
PM의 요구 문장 예시:  
  - “GNB에는 워크스페이스 전환 드롭다운을 포함하고, LNB는 ‘대시보드/공유/분석/설정’ 4개 섹션을 보여주세요.”  
  - “권한이 없는 섹션은 LNB에서 숨기고(또는 비활성), 직접 URL 접근 시 권한 에러 페이지로 라우팅합니다.”

**5) 안티패턴**  
- “메뉴 좀 만들어주세요”처럼 구조/행동이 불명확  
- 고정요소(Sticky) 요구가 개발 난이도를 올리는데 정의 없이 요청  
- 반응형(모바일/웹)에서 GNB/FNB 룰이 없어서 깨짐

**6) 체크리스트**  
- [ ] 공통 요소(헤더/푸터/내비)와 페이지별 요소가 분리되어 있다  
- [ ] 고정/스크롤 행동이 정의돼 있다  
- [ ] 권한/로그인 상태별 메뉴 노출 규칙이 있다

**7) 지표/측정**  
- 탐색 성공률(UT), 메뉴 클릭 분포(히트맵), 이탈률

(구성요소 자체는 업계 관용이지만, “사용자 행동 관찰/히트맵/퍼널”로 검증 가능)
