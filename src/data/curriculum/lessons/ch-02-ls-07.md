## 부록과 참고문헌

| 한글 | 영문(필요 시) | 유사어/혼동 | 흔한 혼동 포인트 |
|---|---|---|---|
| 프로덕트 매니저 | Product Manager | 서비스 기획자 | “문서 담당자”로만 오해 |
| 프로젝트 매니저 | Project Manager | PJM | 가치보다 일정만 보는 역할로 축소되기도 함  |
| 프로덕트 오너 | Product Owner | PO | Scrum에선 ‘가치 최대화’ 책임, 회사에선 직무명으로 변형  |
| PRD | Product Requirements Document | BRD/SRS | “상세 스펙”이 아니라 정렬 문서  |
| IA | Information Architecture | 사이트맵 | 메뉴 트리만이 아니라 라벨/탐색/검색까지 포함  |
| Use Case | Use Case | User story | 예외/대안 흐름을 포함한 시나리오 중심  |
| Wireframe | Wireframe | Mockup/Prototype | 와이어=구조, 목업=비주얼, 프로토타입=인터랙션  |
| User Flow | User Flow | 서비스 플로우 | 결정점/분기/이탈을 포함해 흐름을 나타냄  |
| Journey Map | Journey map | CJM | 행동+생각+감정 기반이어야 함  |
| 퍼널 | Funnel | Conversion funnel | 단계 정의가 핵심(너무 많아도/적어도 문제)  |
| AARRR | Pirate Metrics | 성장 퍼널 | ‘한 번에 다’가 아니라 병목 단계 집중  |
| 활성 사용자 | MAU/DAU/WAU | Active user | Active 정의가 제품마다 다름  |
| LTV/CLV | Customer Lifetime Value | — | 가정(마진/이탈률)을 명시해야 함  |
| UT | Usability test | 사용성 테스트 | 설문이 아니라 과업+관찰  |
| 히트맵 | Heatmap | Click/Scroll/Move map | 세그먼트 없이 평균만 보면 위험  |

- Day 1: PM 역할/책임 정리(내 말로 5문장) + RACI 초안 작성   
- Day 2: AARRR와 퍼널 차이 이해 + 내 서비스(가상) AARRR 지표 1개씩 잡기   
- Day 3: 페르소나 1개 작성(Bad→Good 리비전)   
- Day 4: 저니 맵 표(단계/행동/감정/페인) 작성   
- Day 5: User Flow(결정점 포함) 그리기   
- Day 6: IA 트리 설계(Depth 2~3)   
- Day 7: Wireframe 1장(홈/핵심 플로우)   
- Day 8: Use Case 1개(예외 2개 포함) 작성   
- Day 9: Storyboard 6컷 작성   
- Day 10: PRD 초안 작성(목표/지표/범위/In-Out)   
- Day 11: UT 과업 5개 작성 + 진행 스크립트 초안   
- Day 12: UT 모의(친구 1명) + 관찰 노트 정리   
- Day 13: 히트맵/퍼널 ‘가상 결과’로 PRD 업데이트(변경 문장 5개)   
- Day 14: 포트폴리오 패키징(Problem→Decision→Result 형태로 편집) + 다음 2주 계획
