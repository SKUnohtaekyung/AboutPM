# Ch.4 — Lesson 2/3: 실전 기획: User Flow와 로직 명세 설계

> **학습 위치**: Chapter 4 · 2번째 레슨 / 전체 3개 레슨  
> **이전 단계**: 가설 검증의 중요성과 올바른 타겟팅(PMF 지향)을 배웠습니다.  
> **다음 단계**: 기획의 마지막 단계인 빈 화면과 서버 통신 상태, 그리고 Feasibility(실현 가능성) 검증으로 넘어갑니다.

---

## 학습 목표
- User Flow와 IA의 차이를 이해하고 예외/대체 경로(Alternate Flow)를 설계할 수 있다
- 개발자가 코딩할 수 있는 수준의 화면 로직(디스크립션)을 작성할 수 있다

## 핵심 용어
- **User Flow**: 사용자의 동선을 순서대로 그린 흐름도
- **IA (Information Architecture)**: 서비스의 정보 구조 지도
- **Happy Path**: 사용자가 오류 없이 목표를 달성하는 이상적 경로
- **Alternate/Exception Flow**: 오류 상황이나 대안적 시나리오가 발생했을 때의 예외 경로 (속칭 Unhappy Path)
- **디스크립션**: 화면설계서에서 화면의 동작 로직을 텍스트로 설명한 것

## STEP 2. User Flow 설계하기

기능이 정해지면 사용자가 그 기능을 쓰는 **동선(User Flow)**을 그립니다.

![User Flow - 동선 설계](/generated/inline_ch4.svg)

### User Flow vs. IA — 무엇이 다른가?

| | User Flow | IA (Information Architecture) |
|---|---|---|
| **비유** | 사용자가 걸어다니는 길 | 백화점 층별 안내도 |
| **형태** | 흐름도 (순서가 있는 다이아몬드, 네모 상자) | 트리 구조 (메뉴와 페이지 계층도) |
| **목적** | "목표 달성까지 어떻게 이동하는가?" | "이 정보가 어느 카테고리에 있는가?" |

### Happy Path vs. Alternate / Exception Flow

User Flow를 그릴 때, 이상적인 경로만 그리면 개발 단계에서 무수한 질문이 쏟아집니다.

**1) Happy Path (이상적 경로):**  
사용자가 실수 없이 한 번에 목표를 달성하는 흐름. 기본적으로 누구나 기획하는 경로입니다.

> 예: 로그인 화면 → 이메일 입력 → 비밀번호 입력 → API 검증 성공 → 메인 홈

**2) Alternate / Exception Flow (대안/예외 경로):**  
실무 커뮤니티에서는 흔히 `Unhappy Path`라고도 부릅니다 (UX/UI 표준 용어로는 Alternate 또는 Exception Flow가 더 정확합니다).  
**이걸 챙기는 게 진짜 기획자의 역량입니다.**

> 예: 로그인 화면 → 존재하지 않는 이메일 조회 → "가입되지 않은 이메일입니다" 안내 표출 및 회원가입 버튼 활성화 유도  
> 예: 5회 이상 비밀번호 검증 실패 → 계정 잠금 처리 → 캡차 인증 또는 이메일 재설정 흐름으로 분기

이 질문에 대한 답이 모두 Flow Chart에 그려져 있어야 합니다.

## STEP 3. 상세 설계 — 개발자가 읽는 기획서

흐름도(Flow)가 완성되면 하나의 화면 안에서 벌어지는 일을 텍스트로 풀어씁니다. 이것을 **디스크립션(Description)** 또는 **로직 명세**라고 합니다.

![청사진 - 로직 명세](/generated/blueprint.svg)

### 나쁜 디스크립션 vs. 좋은 디스크립션

❌ **나쁜 예:**
> "버튼을 누르면 다음 화면으로 이동한다."

✅ **좋은 예:**
> "회원가입 완료 버튼을 탭할 때:  
> 1. 이메일/비밀번호 정규식 폼 유효성 검사 (실패 시 → 해당 필드 상태를 Error로 변경 후 사유 표출)  
> 2. 검사 성공 시 중복 이메일 체크 API 호출 (200 OK 응답 수신 시 진행, 409 Conflict 수신 시 토스트 표출)  
> 3. 성공 응답 시 로컬 스토리지에 토큰을 저장하고 메인 홈으로 Routing 한다."

좋은 디스크립션의 3가지 핵심 조건:
1. **조건 분기**가 모두 명시되어 있다 (if/else 상황)
2. **데이터 평가와 API 호출 시점**이 명확하다
3. **오류 시 대응 프로세스**가 포함되어 있다
