# Ch.4 — Lesson 3/3: 기획 완성: 4대 상태 정의와 Feasibility

> **학습 위치**: Chapter 4 · 3번째 레슨 / 전체 3개 레슨  
> **이전 단계**: User Flow와 로직 명세를 익혔습니다.  
> **다음 단계**: 4대 상태 모서리 케이스 방어와 일정/리소스 실현 가능성 점검으로 기획을 방어합니다.

---

## 학습 목표
- 사용자가 직면할 수 있는 4대 상태(Ideal/Empty/Loading/Error)를 설계한다
- 앱/웹 환경에서의 Skeleton UI 개념과 네트워크 로딩 응답 대응 지침을 안다
- 일정과 인력 리소스에 맞춘 기술적 실현 가능성(Feasibility) 및 스펙 아웃 전략을 이해한다

## 핵심 용어
- **4대 상태**: Ideal, Empty, Loading, Error로 구분되는 화면의 주기 상태
- **Skeleton UI**: 콘텐츠가 로드되기 전 실제 레이아웃과 동일한 모양의 회색 뼈대를 렌더링하는 UI
- **Feasibility (실현 가능성)**: "이 기획이 기술적으로 구현 가능하며, 주어진 기간과 인원으로 구축할 수 있는가?"
- **스펙 아웃 (Spec-out)**: 리소스/일정 조건에 따라 덜 중요한 기능을 의식적으로 범위 밖으로 빼는 결정

---

## STEP 1. 4대 상태 정의 — 기획의 완성도 높이기

기획서를 작성할 때 꽉 찬 데이터가 그려진 예쁜 화면 위주로 기획하는 데 그치면 실무에서 수많은 모서리 케이스(Edge Case)를 놓치게 됩니다. 모든 화면은 최소 **4개의 상태**로 기획되어야 합니다.

### 1) Ideal (이상적 상태)
데이터가 꽉 차 있고 모든 것이 완벽하게 작동하는 메인 시안입니다. 가장 흔하게 그리는 화면입니다.

### 2) Empty (빈 상태)
게시글이 0개이거나 검색결과가 없는 상태 등 콘텐츠가 비워져 있을 때입니다.
- ❌ "텅 빈 하얀 화면" (고장난 것으로 오인됨)
- ✅ "아직 작성한 기록이 없네요! 첫 기록을 남겨보세요!" + **행동 유도 CTA 버튼 배치** (온보딩 및 리텐션의 핵심 구간)

### 3) Loading (로딩 상태)
API 등 서버에서 데이터를 불러오는(fetching) 상태입니다.
단순히 모래시계나 빙글빙글 도는 스피너만 두면, 사용자 체감 대기 시간이 더 길게 느껴집니다.

**💡 실무 개념: Skeleton UI (스켈레톤 UI)**
> 콘텐츠 자리의 윤곽(Skeleton)을 실제 그 페이지의 레이아웃과 똑같은 모양의 회색 박스/줄로 먼저 렌더링해 놓는 기법입니다. *(가상 배경을 흐릿하게 만드는 Blur Placeholder와는 다릅니다)*
> 사용자는 빈 화면을 보며 기다리는 대신 "곧 이 뼈대 자리에 데이터가 나타나겠구나"라고 무의식적으로 안심하여 체감 대기 속도가 줄어듭니다.

**네트워크 통신 대기 규칙:**
- **즉각 피드백**: 일반적으로 2~3초 이상 응답이 없으면 사람들은 앱이 멈췄다고 생각합니다.
- **처리 전략**: 아주 짧은 시간 내의 연산은 인디케이터나 스켈레톤을 활용하고, 백엔드의 대규모 쿼리나 파일 업로드 같은 장시간의 작업은 사용자를 묶어두는 화면 대신 **"백그라운드 처리 + 푸시/인앱 알림 통지"**로 기획해야 합니다.

### 4) Error (에러 상태)
권한 부족, 네트워크 끊김, 500 인터널 서버 오류 등 예상 가능한 모든 에러입니다.
- ❌ "알 수 없는 에러가 발생했습니다."
- ✅ "서버와 연결이 끊어졌어요. Wi-Fi 상태를 확인해보세요." + **[새로고침] 버튼이나 대안 경로 제시**

## STEP 2. 개발 착수 전: Feasibility 확인

PM이 기획서를 개발자에게 전달할 때 가장 중요한 관문이 **Feasibility(실현 가능성)** 검증입니다. 

> ⚠️ 주의: 초심자들은 Feasibility를 "기획서가 잘 읽히나?"의 문서 표현력 문제로 착각하지만, 진짜 Feasibility는 **"기술적 구현 가능성"**과 **"제한된 리소스(시간/자원) 내 통과 여부"**의 결속성입니다.

- "이 복잡한 애니메이션들을 쓸 때 우리 프로젝트 성능에 영향을 주지 않나요?"
- "이 기능을 구현하려면 외부 결제 API의 심사 등 3주가 묶이는데, 우리는 2주 안에 오픈해야 합니다. 어떻게 할까요?"

개발자의 이 질문들에 조율하면서 합의하는 과정이 바로 Feasibility 점검입니다.

### 스펙 아웃 (Spec-out)과 우선순위 관리
점검 결과 일정이 초과될 것 같다면 기능 목록(Feature List)을 쳐내야 합니다.
개발자가 알아서 짜를 기능을 임의로 정하게 두면 안 됩니다. PM이 미리 **우선순위(Prioritization)**를 쥐고 조율해야 합니다.

* (스타트업 등에서 흔히 쓰는 간단한 분류 예시, MoSCoW 방법론을 병행하기도 합니다)
1. **P0 (Critical)**: 이게 안 돌아가면 우리 앱은 아예 핵심 가설을 시험조차 못 한다 (반드시 이번 배포에 사수)
2. **P1 (Important)**: 가능하면 들어가야 경쟁력을 얻지만 기술/일정의 블로커가 심각하다면 대체나 우회법을 논의한다 (우회 방안 모색)
3. **P2 (Nice-to-have)**: 있으면 좋으나 굳이 급하지 않다 (과감히 **스펙 아웃**)

> 기획자는 더하는(Add) 사람이지만, 매니저(PM)는 상황에 따라 핵심을 위해 포기할(Drop) 것을 결정하는 사람이라는 것을 잊지 마세요.
