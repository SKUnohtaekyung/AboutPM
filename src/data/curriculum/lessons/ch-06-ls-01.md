## 학습 목표
- 실전 SQL 기초, 이벤트 로깅(Taxonomy), 코호트 분석을 배운다


---

과거의 PM은 사용자의 피드백을 수집하고 화면 기획서를 그리는 데 집중했습니다. 그러나 현대의 제품 환경은 거대한 데이터를 뿜어내고 있으며, 인공지능(AI)은 소프트웨어의 규칙마저 파괴적으로 혁신하고 있습니다. 누군가 만들어놓은 대시보드만 수동적으로 바라보거나, 챗GPT를 단순한 검색 도구로만 활용하는 PM은 도태될 수밖에 없습니다.
본 챕터에서는 데이터 엔지니어 없이도 스스로 지표를 추출하는 '데이터 리터러시 하드스킬'과, AI가 뒤바꿔놓을 '제품 기획의 패러다임에서 당장 실천해야 할 생존 전략'을 다룹니다.

모든 데이터 분석의 시작점인 AARRR(해적 지표)과 퍼널 구조를 이해했다면, 이제는 현실의 데이터베이스와 트래킹 툴에서 나에게 필요한 데이터를 직접 설계하고 추출할 수 있는 하드스킬이 필요합니다.

대규모 스타트업이나 테크 기업이 아닌 이상, 필요할 때마다 데이터 분석가에게 일일이 SQL 추출(Extract)을 부탁하는 것은 업무 속도를 심각하게 지연시킵니다. PM은 `SELECT, FROM, WHERE, GROUP BY, JOIN` 등 핵심 구문 중심의 기본 SQL 문법을 습득해야 합니다. 
핵심은 완벽한 쿼리를 작성하는 것이 아니라, "우리 제품의 사용자 행동 로그가 User 테이블과 Order 테이블에 어떻게 나누어져 저장되어 있는지(스키마 스파게티의 형태)"를 머릿속에 그릴 수 있는 능력을 갖추는 것입니다.

퍼널이나 히트맵 분석 도구(Amplitude, Hotjar 등)는 자동으로 모든 데이터를 모아주지 않습니다. PM은 백엔드 DB에 쌓이는 '결과/상태 데이터(Schema)'와, Amplitude 등에 쏘는 프론트엔드 '행동 로그 데이터(Taxonomy)'를 명확히 구분해서 설계해야 합니다. 사전에 기획서(PRD)에 "사용자가 장바구니 버튼을 눌렀을 때, `click_add_cart`라는 이벤트를 로깅하고, 파라미터로 `item_category`와 `price`를 함께 전송해달라"고 명확히 (Taxonomy) 설계하는 것은 후자에 속합니다. 잘못된 로깅 설계는 "결제 버튼은 많이 눌렸는데, 대체 어떤 상품을 담은 유저가 결제했는지 맵핑할 수 없는" 데이터 쓰레기를 낳습니다.

![데이터 온실 - 코호트 분석과 데이터 설계](/generated/data_greenhouse.svg)

모든 전체 사용자를 뭉뚱그려 분석하는 것은 '평균의 함정'에 빠지는 길입니다. "지난주 금요일 할인 이벤트에 반응하여 가입한 유저 집단"과 "어제 자발적 검색을 통해 가입한 유저 집단"을 각각 묶어(Cohort), 이들이 시간이 지남에 따라 리텐션 추이가 어떻게 다르게 나타나는지 분리하여 추적해야 진정한 인사이트가 도출됩니다.
